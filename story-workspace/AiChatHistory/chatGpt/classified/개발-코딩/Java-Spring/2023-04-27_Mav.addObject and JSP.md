---

**ChatGPT**

`mav.addObject()`은 Spring MVC에서 사용되는 메서드로, 뷰에서 사용될 데이터를 `ModelAndView` 객체에 추가하는 데 사용됩니다. `ModelAndView` 객체는 Spring MVC의 모델에 해당하며, 뷰에 전달되어 렌더링 됩니다.

`mav.addObject()`를 사용하여 추가한 데이터는 JSP에서 EL (Expression Language)을 사용하여 가져올 수 있습니다. 이때 사용되는 스코프는 기본적으로 request입니다.

Session은 서버 측에서 유지되는 상태 정보를 저장하는 객체입니다. `mav.addObject()`를 사용하여 전달한 데이터는 JSP에서 세션에 저장되지 않으며, 뷰에서만 사용됩니다. 따라서 `mav.addObject()`를 사용하여 전달한 데이터는 세션과는 별개의 개념입니다.

그런데 만약 mava.addObject랑 session.setAttribute로 선언한 객체의 이름이 같다면 값이 어떻게 불러져? #{value} 이런식으로 가져오면

---

**ChatGPT**

`mav.addObject()`와 `session.setAttribute()`로 선언한 객체의 이름이 같다면, JSP에서 EL(Expression Language)을 사용하여 값을 가져올 때 충돌이 발생할 수 있습니다. 

EL은 데이터를 가져올 때 다음과 같은 순서로 검색합니다.

1. PageContext
2. Request
3. Session
4. Application

즉, 먼저 `PageContext`에서 찾고, 없으면 `Request`, `Session`, `Application` 순서로 검색합니다. 따라서 동일한 이름을 갖는 객체가 `Request`와 `Session`에 모두 존재하는 경우, `Request` 객체의 값을 우선하여 가져옵니다.

예를 들어, `mav.addObject("data", "value1")`와 `session.setAttribute("data", "value2")`를 사용하여 `data`라는 이름의 객체를 각각 추가한 경우, JSP에서 `${data}`를 사용하여 값을 가져오면 `Request` 객체에서 `value1`을 우선으로 가져옵니다. 

하지만 이러한 경우를 방지하기 위해 객체 이름을 유니크하게 만들거나, 객체의 스코프(scope)를 명시적으로 지정하여 값을 가져오는 것이 좋습니다. 예를 들어, `mav.addObject("request_data", "value1")`와 `session.setAttribute("session_data", "value2")`와 같이 이름을 다르게 하거나, `${sessionScope.data}`와 같이 스코프를 명시적으로 지정하여 값을 가져올 수 있습니다.

JPA는 **데이터베이스의 테이블과 객체를 매핑**하는 방식입니다. JPA의 엔티티 클래스는 데이터베이스와 직접적으로 연결되며, 데이터를 **영속적으로 관리**합니다.

### JPA와 DTO의 관계:
JPA는 **데이터베이스와의 상호작용**을 담당하고, **DTO는 데이터를 외부로 전송하는 역할**을 합니다. 그 둘은 주로 **데이터 변환 및 매핑**을 통해 협력합니다. 예를 들어, 데이터베이스에서 조회한 엔티티를 DTO로 변환하여 클라이언트에 전달하는 형태로 사용할 수 있습니다.

### 1. **JPA와 DTO의 기본 차이점**

- **JPA**:
  - JPA는 **ORM (Object-Relational Mapping)**으로, **객체와 데이터베이스 테이블 간의 매핑**을 담당합니다.
  - JPA는 주로 **하나의 테이블에 대한 CRUD 작업**을 처리하는 데 최적화되어 있습니다. 기본적으로 **엔티티 객체**를 통해 데이터를 조회하고 수정합니다.
  - **복잡한 쿼리**나 **조건이 여러 개인 조회**를 위해서는 `JPQL (Java Persistence Query Language)`이나 **Criteria API**를 사용하여 쿼리를 작성할 수 있습니다.
  - **JPA 엔티티**는 **하나의 테이블에 대응**하므로, **단순한 조회**에서부터 **연관 관계가 있는 테이블**까지 자연스럽게 매핑할 수 있습니다.
  - JPA의 주요 장점은 **객체지향적인 접근**과 **자동화된 데이터베이스 관리**입니다. 데이터베이스를 신경 쓰지 않고 객체만 다루면 되기 때문에 생산성이 높습니다.

- **DTO (Data Transfer Object)**:
  - DTO는 **데이터를 전송하기 위한 객체**입니다. 일반적으로 **비즈니스 로직을 포함하지 않고**, **데이터를 특정 형태로 변환**하여 **계층 간 전달**하는 역할을 합니다.
  - DTO는 주로 **복잡한 데이터 조회**나 **여러 테이블에서 가져온 데이터를 한꺼번에 전송**할 때 사용됩니다. 예를 들어, 여러 테이블의 데이터를 **조인**하거나, **집계 함수** 등을 사용해서 데이터를 한 번에 모아서 반환할 때 DTO를 사용합니다.
  - DTO를 사용하면 **특정 뷰**나 **클라이언트에서 필요로 하는 데이터**만 선택적으로 전달할 수 있습니다. 예를 들어, 하나의 엔티티에서 불필요한 필드를 제외하거나, 여러 엔티티의 데이터를 결합하여 **단일 객체로 반환**하는 경우 DTO를 사용합니다.

### 2. **JPA에서 복잡한 조회**

JPA는 기본적으로 **단일 테이블 조회**에 특화되어 있지만, **복잡한 쿼리**를 작성하기 위한 기능도 제공합니다. **JPQL (Java Persistence Query Language)**이나 **Criteria API**를 사용하면 여러 테이블을 **조인**하거나 **집계 함수** 등을 사용할 수 있습니다. 이때 **DTO**를 활용하여 **결과를 반환**할 수 있습니다.

#### 예시: JPQL을 사용하여 복잡한 쿼리 처리

가정: `Order`와 `Product` 테이블이 있고, 특정 조건에 맞는 주문 정보를 조회하고 그 정보를 DTO로 반환하고 싶다고 합시다.

1. **DTO 클래스**:
   ```java
   public class OrderDTO {
       private Long orderId;
       private String productName;
       private Integer quantity;
       private Double price;

       // Constructor, Getters, Setters
   }
   ```

2. **JPA Repository에서 JPQL을 사용하여 조회**:
   ```java
   @Repository
   public interface OrderRepository extends JpaRepository<Order, Long> {

       @Query("SELECT new com.example.OrderDTO(o.id, p.name, oi.quantity, oi.price) " +
              "FROM Order o " +
              "JOIN o.orderItems oi " +
              "JOIN oi.product p " +
              "WHERE o.customer.id = :customerId")
       List<OrderDTO> findOrdersByCustomerId(@Param("customerId") Long customerId);
   }
   ```

이 예제에서는 `Order` 엔티티와 `OrderItem`을 **조인**하고, `Product` 엔티티의 데이터를 **DTO로 변환**하여 반환합니다. **JPQL**을 사용하면 **복잡한 조건**을 처리하면서도 **DTO로 데이터를 전송**할 수 있습니다.

### 3. **DTO 사용과 JPA의 차이점**

- **DTO의 목적**은 **특정 데이터를 전달**하는 것입니다. 즉, 데이터베이스에서 여러 테이블을 **조인하거나 집계**한 후 그 결과를 **DTO로 변환**하여 응답하는 것입니다.
- **JPA 엔티티**는 데이터베이스의 테이블과 1:1로 매핑되며, **CRUD 작업**과 **엔티티 간 관계**를 관리합니다. 복잡한 데이터 처리나 조합이 필요할 때는 **JPQL**을 사용하여 원하는 데이터를 조회하고, 이 데이터를 DTO로 변환하여 전달합니다.
- JPA는 기본적으로 **영속성 관리**에 초점이 맞춰져 있기 때문에, **상태 변화 추적**, **연관 관계 관리**, **캐시 관리** 등도 자동으로 처리됩니다.
- 반면, **DTO는 응답 전용 객체**로 주로 **데이터를 잘라서 필요한 부분만 전달**하는 데 유용합니다.

### 4. **DTO와 JPA 사용 시의 의도 차이**

- **DTO를 사용**하는 경우는 보통 **비즈니스 로직**을 포함하지 않고 **특정 데이터만을 포장**하여 전달하는 목적입니다. 예를 들어, 클라이언트에게 필요한 정보만 포함된 **API 응답**을 생성하는 데 사용됩니다.
  
- **JPA 엔티티를 사용하는 경우**는 주로 **데이터베이스와의 상호작용**이 필요할 때 사용되며, **전체 테이블 데이터**를 직접 조회하거나 **엔티티의 상태를 관리**합니다.

### 5. **DTO와 JPA 엔티티를 함께 사용하는 경우**

실제로 많이 사용되는 방법은 **DTO와 JPA 엔티티를 함께 사용**하는 것입니다. 복잡한 쿼리나 여러 테이블에서 데이터를 조회한 후, 이 데이터를 **DTO로 변환**하여 클라이언트나 다른 계층에 전달하는 방식입니다.

#### 예시:

1. **JPA 엔티티**로 데이터를 조회하고:
   - 여러 테이블을 조인하고, 필요한 데이터만 선택하여 DTO로 반환합니다.
   
2. **DTO 변환**:
   - 복잡한 데이터를 **DTO로 변환**하여 **응답**합니다.

이 방식은 **효율적**이며, **데이터베이스와 객체 간의 매핑**을 유지하면서도 **필요한 데이터만을 깔끔하게 전달**할 수 있는 방법입니다.

### 결론

- **JPA**는 주로 **하나의 테이블에 대한 CRUD 작업**을 자동화하며, **영속성 관리**와 **상태 추적**을 담당합니다.
- **DTO**는 주로 **데이터를 전송**하기 위한 객체로, 복잡한 조회 결과를 **간결하게 담아서 전달**하는 데 사용됩니다.
- **JPA**와 **DTO**는 **상호 보완적**으로 함께 사용될 수 있습니다. **JPA 엔티티**로 **데이터를 조회하고**, 그 결과를 **DTO로 변환**하여 **전송**하는 방식이 일반적입니다.

### JPA를 사용할 이유

1. **객체지향적 접근**
   - JPA는 **ORM (Object-Relational Mapping)** 기술로, **객체 지향적인 방식**으로 데이터를 처리할 수 있습니다. 즉, **SQL 쿼리 대신 객체**를 다루므로, **테이블에 맞는 객체**를 사용하여 **비즈니스 로직을 작성**합니다. 
   - XML 기반의 CRUD에서는 **SQL 문법을 직접 작성**해야 하지만, JPA에서는 **엔티티 객체**만 다루면 됩니다. 예를 들어, 데이터를 삽입하고 수정할 때도 객체를 다루는 것처럼 처리할 수 있습니다.

2. **자동화된 데이터베이스 작업**
   - **엔티티 객체**를 JPA가 자동으로 **데이터베이스 테이블과 매핑**하여 데이터베이스와의 상호작용을 자동화합니다. 따라서 **SQL 작성량**이 줄어들고, **변경 사항 관리**가 쉬워집니다.
   - 예를 들어, **insert**, **update**, **delete** 작업은 JPA가 알아서 처리해 줍니다. XML에서 이러한 작업을 수동으로 작성해야 하는 번거로움을 줄일 수 있습니다.

3. **복잡한 연관 관계 처리**
   - JPA는 **연관 관계 (OneToMany, ManyToOne, ManyToMany 등)**를 쉽게 처리할 수 있도록 돕습니다. 예를 들어, **객체 간 관계를 설정하고, 연관된 데이터를 가져오는 쿼리**를 직접 작성하지 않아도 됩니다.
   - XML에서 연관 관계를 처리하려면, **조인 쿼리**를 직접 작성하고, 연관된 데이터를 **매핑**해야 하기 때문에 코드가 복잡해질 수 있습니다.

4. **데이터베이스 벤더 독립성**
   - JPA는 **벤더 독립적**입니다. 즉, 데이터베이스의 종류가 바뀌더라도 대부분의 경우 **JPA 설정만으로 처리**할 수 있습니다. 예를 들어, **MySQL**에서 **PostgreSQL**로 바꿀 때 JPA는 거의 **변경이 필요 없으며**, 설정만 수정하면 됩니다.
   - XML 기반으로 SQL을 작성할 경우, **데이터베이스마다 다른 쿼리 문법**을 사용할 수 있기 때문에 **DB 벤더에 의존적**이 됩니다. 이 점에서 JPA가 더 유리할 수 있습니다.

5. **영속성 관리 (Persistence Context)**
   - JPA는 **영속성 컨텍스트**를 통해 **엔티티의 상태를 관리**합니다. 즉, 동일한 엔티티 객체를 여러 번 조회할 때, **첫 번째 조회 시 데이터베이스에서 데이터를 가져오고**, 그 후에는 **메모리에서 객체를 재사용**합니다. 이는 **성능 향상**에 도움을 줍니다.
   - XML에서는 이와 같은 **영속성 관리**를 별도로 구현해야 하며, **세션 관리**를 신경 써야 합니다.

6. **쿼리 최적화**
   - JPA는 **JPQL** (Java Persistence Query Language)과 **Criteria API**를 제공하여 **객체 기반 쿼리**를 작성할 수 있습니다. 또한 **네이티브 쿼리**를 사용할 수 있어, 필요할 경우 SQL을 직접 작성할 수도 있습니다.
   - XML에서는 SQL을 직접 작성해야 하기 때문에, 쿼리의 성능 최적화를 위해 **수동으로 최적화**를 해야 합니다. JPA에서는 일부 **쿼리 최적화**를 자동으로 처리하는 경우도 있습니다.

7. **트랜잭션 관리**

### XML 기반 CRUD의 장점

- **세밀한 제어**: XML에서 직접 **SQL 쿼리**를 작성하는 것은 매우 **직관적**이고, 복잡한 쿼리나 최적화가 필요한 경우에는 **세밀하게 제어**할 수 있는 장점이 있습니다.
- **경량화**: JPA와 같은 **ORM**은 초기 학습 비용이 있을 수 있지만, XML 기반에서는 **복잡한 설정 없이 바로 SQL 작업**을 할 수 있습니다. 특히 **간단한 CRUD 작업**에서는 성능 면에서 더 유리할 수 있습니다.
- **성능 최적화**: **수동으로 작성한 SQL**을 사용하면, **필요한 쿼리만 작성**하여 성능을 **세밀하게 최적화**할 수 있습니다. 반면 JPA에서는 자동 생성된 SQL에 의존하기 때문에 최적화가 어려운 경우도 있습니다.

### JPA를 도입할지 말지 결정하는 기준

1. **프로젝트의 규모와 복잡도**:

2. **유지보수의 용이성**:
   - JPA는 **객체 기반**으로 설계되기 때문에, **코드의 가독성**과 **유지보수성**이 높아집니다. 또한 **엔티티 클래스를 수정**하면, 데이터베이스와의 연관이 자동으로 처리되므로 **변경 관리가 용이**합니다.
   - XML에서는 **SQL 쿼리**를 수동으로 작성하고, 매번 **매핑 설정**을 변경해야 하므로 유지보수가 복잡해질 수 있습니다.

3. **학습 비용**:
   - JPA는 **추상화**가 많기 때문에 처음에는 **학습 비용**이 있을 수 있습니다. 하지만 **중장기적으로** 보면 생산성 향상에 도움이 될 수 있습니다.
   - XML 기반으로 CRUD를 처리하는 방식은 **직관적**이고, **즉시 적용**할 수 있어 초기 학습 비용이 적습니다.

### 결론

- **JPA를 쓸 이유가 있냐?**는 질문에 대한 답은, 프로젝트의 **규모와 요구 사항**, **유지보수성**, **객체 지향 설계** 등이 중요한 요소입니다. 만약 기존에 XML을 잘 사용하고 있고, **간단한 CRUD 작업만 필요**하다면 JPA를 도입할 필요가 없을 수도 있습니다. 하지만 **객체 지향적인 설계**와 **자동화된 데이터베이스 관리**가 필요한 **대규모 프로젝트**라면 JPA가 유리할 수 있습니다.

### JPA 사용 시 장단점

#### 장점
1. **연관 관계 처리**:
   - JPA는 **연관 관계**를 처리하는 데 강점을 가집니다. `@OneToMany`, `@ManyToOne`, `@ManyToMany`와 같은 **어노테이션**을 통해 데이터베이스 테이블 간 관계를 **객체 지향적으로 처리**할 수 있습니다. 
   - 예를 들어, 두 테이블을 조인하는 것이 아닌 **객체 간 관계**를 정의하여 JPA가 이를 처리하도록 할 수 있습니다. 

2. **영속성 관리**:
   - JPA는 **영속성 컨텍스트**를 통해 데이터를 관리하고, 동일한 엔티티를 여러 번 조회할 때 **메모리에서 재사용**합니다. 따라서 쿼리 성능에 대한 걱정을 어느 정도 덜 수 있습니다.
   - **하이버네이트**와 같은 JPA 구현체는 **쿼리 최적화**를 어느 정도 자동으로 해주기 때문에, 복잡한 쿼리문을 직접 관리하지 않아도 됩니다.

3. **자동화된 데이터베이스 변경 관리**:
   - JPA에서는 엔티티 객체를 통해 데이터베이스에 저장된 정보를 **자동으로 매핑**하고, 데이터 변경 사항이 있을 때 **객체 상태 변경**만으로 데이터베이스도 반영됩니다. 
   - 예를 들어, **insert**, **update**, **delete** 작업을 **엔티티 객체의 상태 변경**만으로 처리할 수 있기 때문에 **수동으로 쿼리를 작성할 필요**가 없어지므로 유지보수성이 높습니다.

4. **쿼리 최적화 기능**:
   - JPA는 **JPQL** (Java Persistence Query Language) 및 **Criteria API**를 사용하여 객체 지향적인 쿼리를 작성할 수 있게 도와줍니다. 또한, **네이티브 쿼리**도 지원하여 **성능 최적화가 필요한 쿼리**는 직접 작성할 수 있습니다.

#### 단점
1. **복잡한 쿼리 처리**:
   - **JOIN문**과 같은 **복잡한 쿼리**가 많이 필요한 경우, JPA는 다소 **비효율적일 수 있습니다**. 특히 여러 테이블을 **조인**해야 하는 경우, JPA가 생성하는 SQL 쿼리가 **예상보다 복잡하거나 비효율적**일 수 있습니다.
   - 또한, JPA는 복잡한 **동적 쿼리 생성**에 제한이 있을 수 있어, **유연한 쿼리**가 필요한 경우에는 직접 **네이티브 SQL**을 사용하는 것이 더 효율적일 수 있습니다.

2. **학습 곡선**:
   - JPA는 **객체 지향적 설계**가 기본이기 때문에, 처음 시작할 때 **학습 곡선이 존재**합니다. 또한, 엔티티 설계나 **연관 관계 매핑**에서 실수할 수 있기 때문에 **주의가 필요**합니다.

3. **쿼리 성능 문제**:
   - 복잡한 쿼리에서 **N+1 문제**와 같은 성능 문제가 발생할 수 있습니다. 이를 해결하려면 **FetchType.LAZY**, **FetchType.EAGER** 등 **연관 관계 로딩 전략**을 신중하게 선택해야 합니다.
   - 또한, **쿼리 최적화**를 위해 `@Query` 어노테이션을 통해 **JPQL**을 작성할 때, SQL에 비해 성능이 떨어질 수 있는 부분이 있을 수 있습니다.

### DTO 활용 방식

#### 장점
1. **명시적 제어**:
   - **DTO**를 사용하면 **SQL을 명시적으로 작성**하고, **데이터 매핑을 직접 제어**할 수 있습니다. 즉, 어떤 데이터를 가져올지, 어떤 조인문을 사용할지 **세밀한 제어**가 가능합니다.
   - **복잡한 JOIN 쿼리**가 필요할 때, XML 파일에서 직접 **복잡한 SQL**을 작성할 수 있어, **복잡한 조건**이나 **동적 쿼리**에 유리합니다.

2. **성능 최적화**:
   - 직접 쿼리를 작성하는 방식이므로, **쿼리 최적화**가 **직접적으로 가능**합니다. 예를 들어, `INNER JOIN`, `LEFT JOIN`을 세밀하게 조정하거나, **인덱스 사용**, **서브쿼리** 등을 고려하여 성능을 최적화할 수 있습니다.
   - 복잡한 **조인문**을 여러 개 사용하거나, **대용량 데이터**를 처리할 때, **필요한 컬럼만 선택**하거나, **필요한 데이터만 조인**할 수 있어 **성능을 높일 수 있습니다**.

3. **SQL 유연성**:
   - **XML 매핑 파일**에서는 **동적 SQL**을 자유롭게 작성할 수 있기 때문에, **사용자가 정의한 조건에 맞는 SQL**을 유연하게 만들 수 있습니다. 이로 인해 **복잡한 조건문**이나 **다양한 필터링**이 필요한 경우 매우 유리합니다.

#### 단점
1. **복잡한 매핑 관리**:
   - **DTO 매핑**은 JPA보다 더 많은 **수동 작업**이 필요합니다. 예를 들어, **쿼리에서 반환한 결과를 DTO로 매핑**하려면 별도의 **매핑 로직**을 작성해야 하며, 이는 **추가적인 코드**를 필요로 합니다.
   - **DTO 객체가 많아지면** 그에 맞는 **매핑 로직**도 늘어나므로, 코드가 **복잡해질 수 있습니다**.

2. **자동화 부족**:
   - DTO를 사용하면 데이터베이스 변경에 따라 **매핑 로직을 직접 수정**해야 하므로, 데이터베이스 스키마가 자주 변경된다면 매번 DTO와 SQL 쿼리의 변경 사항을 맞춰줘야 하는 불편함이 있을 수 있습니다.
   - JPA에 비해 **자동화된 관리**가 부족합니다.

3. **유지보수 어려움**:
   - SQL을 **직접 작성**하고, **DTO 매핑**을 수동으로 관리해야 하므로, **복잡한 데이터 구조**에서는 **유지보수가 어려워질 수 있습니다**. 또한, **쿼리 수정이 많을 경우** 매번 코드 수정이 필요하고, **일관성 유지**가 어려울 수 있습니다.

### 결론: JPA vs DTO 활용 방식

1. **DB 변경 사항이 자주 발생하고 복잡한 JOIN 쿼리가 필요한 경우**:
   - **DTO 활용**이 더 적합할 수 있습니다. DTO와 XML에서 직접 쿼리를 작성하는 방식이 **유연하고 세밀한 제어**가 가능하기 때문에, **복잡한 쿼리**와 **빈번한 DB 변경**에 더 적합합니다. DTO를 사용하면 **변경 사항을 반영하기 쉬우며**, 복잡한 쿼리문을 **동적으로 조정**할 수 있습니다.

2. **객체 지향적인 접근과 자동화된 데이터 처리**가 필요하다면:
   - **JPA**가 더 적합할 수 있습니다. JPA는 **자동화된 데이터 처리**와 **객체 지향적인 설계**를 제공합니다. 하지만 **복잡한 JOIN문**과 같은 **고급 쿼리**에 대해서는 **네이티브 쿼리**나 **JPQL**을 사용해 최적화할 수 있으며, **유지보수성**도 높아질 수 있습니다.

따라서 **DB 변경이 많고 복잡한 쿼리가 필요하다면 DTO 방식이 유리**할 수 있으며, **자동화와 객체 지향적 설계**가 필요한 경우 **JPA**가 유리할 수 있습니다. **혼합 사용**도 가능합니다. 예를 들어, JPA를 사용하되, **복잡한 쿼리**가 필요한 부분만 **DTO와 XML**을 사용하여 처리하는 방식으로 접근할 수 있습니다.
