## 📌 소프트웨어 개발 방법론 개요
소프트웨어 개발 방법론은 **소프트웨어를 어떻게 계획·설계·구현·테스트·배포할지**에 대한 일련의 프로세스와 규칙을 정의한 체계예요. 각 방법론은 시대적 배경, 조직 규모, 프로젝트 특성에 따라 장단점이 있습니다.  

---

### 1. 전통적 방법론 (계획 기반)
#### ✅ 폭포수 모델 (Waterfall Model)
- 특징: 단계별 순차 진행 (요구사항 → 설계 → 구현 → 테스트 → 유지보수)
- 장점: 명확한 문서화, 관리가 용이
- 단점: 초기에 요구사항이 잘못되면 뒤집기 어렵고, 변경에 취약
- 쓰임새: 정부·국방·금융권 등 규제가 많고 변경이 적은 대규모 프로젝트  

#### ✅ V-모델
- 폭포수의 변형, 각 개발 단계와 테스트 단계를 1:1 매칭
- 장점: 테스트 중시, 결함 조기 발견
- 단점: 역시 변경에 약함  

---

### 2. 반복적/점진적 방법론
#### ✅ 나선형 모델 (Spiral Model)
- 위험 관리 중심, 반복 주기마다 요구사항·설계·구현·평가를 수행
- 장점: 위험 분석에 강함, 점진적 완성 가능
- 단점: 복잡하고 관리 비용이 큼  

#### ✅ 프로토타이핑 모델
- 빠르게 시제품(프로토타입)을 만들어 피드백
- 장점: 사용자 요구 확인이 쉬움
- 단점: 임시 코드가 남아 품질 저하 위험  

---

### 3. 애자일 이전의 대안적 방법론
#### ✅ RAD (Rapid Application Development)
- 빠른 개발과 사용자 참여 강조
- 프로토타입 + 반복 개발 방식
- 장점: 개발 속도 ↑, 사용자의 의견 반영 용이
- 단점: 대규모/복잡 프로젝트에는 한계  

#### ✅ RUP (Rational Unified Process)
- 유스케이스 기반, 반복적·점진적 개발
- 4단계(시작–정교화–구축–전환)로 나눔
- 장점: 명확한 산출물, 대규모 프로젝트에 적합
- 단점: 과정이 복잡하고 도입 비용이 큼  

---

### 4. 애자일 (가볍게만 짚고 패스 👋)
- 스크럼, XP, 칸반 등이 대표적
- “변화 수용 + 협업 + 짧은 주기” 강조
- 스타트업, 중소형 팀에 유리  

---

### 5. 최근/특수 방법론
#### ✅ DevOps
- 개발(Dev)과 운영(Ops)을 통합 → 자동화, 지속적 배포(CI/CD)
- 장점: 빠른 배포와 피드백
- 단점: 조직 문화 변화 필요  

#### ✅ MDD (Model-Driven Development)
- 모델 중심 개발, 코드 자동 생성 강조  
- 장점: 추상화 ↑, 생산성 향상
- 단점: 툴 의존성 크고 학습 비용 ↑  

#### ✅ 테스트 주도 개발(TDD), 행동 주도 개발(BDD)
- 코드 작성 전 테스트 케이스 작성
- 품질 확보에 강점  

---

## 📝 정리
- **전통적 방법론** → 계획·문서 중시, 변경에 약함  
- **반복적/점진적** → 위험 관리, 사용자 참여, 점진적 완성  
- **RAD, RUP** → 애자일 이전 대안, 유연성 확보 노력  
- **애자일** → 가볍고 유연, 협업 중심  
- **현대적 방법론** → DevOps, TDD 등 자동화·품질·배포 속도 강조

## 🚧 네 상황과 가까운 개발 스타일
### 1. **Ad-hoc 개발 (임시방편식)**
- 말 그대로 **즉흥적이고 계획 없는 개발**.  
- 설계 문서 없음 → 코드가 설계 그 자체.  
- 계획도 수시로 바뀌고, 개발자가 알아서 때려넣는 식.  
- 장점: 속도(?)는 나름 빠를 수 있음.  
- 단점: 유지보수 지옥, 버그 양산, 새 개발자가 합류하면 적응 어려움.  

---

### 2. **코드 앤 픽스(Code and Fix)**
- “코드를 그냥 짜고, 문제가 생기면 땜빵하는 방식”.  
- 초기 PC 소프트웨어 시절 많이 쓰였던 원시적인 접근.  
- 특징: 설계/계획 최소화, 완전히 “빨리 만들어서 굴려보자” 느낌.  
- 단점: 시간이 갈수록 시스템이 무너짐. 결국 재개발 필요.  

---

### 3. **카우보이 코딩(Cowboy Coding 🤠)**
- 개발자가 “내가 보스다” 하면서 자유롭게 코딩.  
- 설계도 문서도 팀 규칙도 없이, 그냥 **개인 역량에 올인**.  
- 스타트업 초반, 사이드 프로젝트, 학부생 팀플 같은 데서 자주 발생.  
- 장점: 창의적이고 빠를 수 있음.  
- 단점: 확장 불가능, 인원 늘어나면 바로 폭망.

## 🎮 소규모(1~3인) 게임 개발의 현실적인 특징

### 1. **문서보다 머릿속 설계**
- 기획서나 설계 문서보단, 팀원 머릿속에만 구상이 있음.  
- “이런 느낌으로 만들어보자” → 바로 코드나 에디터로 구현.  
- 개발 중간에 아이디어가 바뀌면 문서 수정보다 그냥 코드를 바꿈.  
- 결과적으로 **Ad-hoc + 프로토타이핑** 방식에 가까움.  

---

### 2. **프로토타입 중심 개발**
- 게임은 감각적인 요소(조작감, 타격감, 밸런스 등)가 중요해서  
  → 종이 설계보다 **직접 돌려보는 게 빠름.**  
- 그래서 초기에 문서 대신 프로토타입을 계속 만들어보면서  
  “재밌냐?”만 검증하는 식으로 진행.  
- 이건 **프로토타이핑 모델**이나 **RAD**에 가깝다고 볼 수 있어요.  

---

### 3. **계획은 유동적**
- 규모가 작아서 의사결정이 빠름 → 오늘 얘기한 게 내일 바뀌는 건 기본.  
- 개발 도중 “이건 재미없다” 하면 그냥 갈아엎는 일도 흔함.  
- 사실상 **유연한 카우보이 코딩 + 반복적 실험**의 혼종.  

---

### 4. **문서화가 없진 않지만, 최소한으로**
- Trello, Notion, Discord, Google Docs 같은 툴로  
  “해야 할 일”, “버그 목록”, “대충의 시스템 설계”만 간단히 적어둠.  
- 즉, **실행 가능한 최소한의 관리**만 하는 수준.  
- 나중에 스케일이 커질 때 그걸 기반으로 프로세스를 확장함.

## 🧠 원래 뜻 vs 판교식 “애자일하게 가자”
| 구분 | 원래의 애자일 | 판교식 애자일 |
|------|----------------|----------------|
| 의미 | 짧은 주기로 결과물 만들고, 피드백 받아서 개선 | 일단 만들어보고 문제 생기면 나중에 고치자 |
| 목표 | 변화에 빠르게 대응하며 품질 유지 | 빨리 데모/서비스 내서 보고하자 |
| 계획 | 매 스프린트(2~4주 단위)마다 구체적 목표 수립 | 오늘 정한 거 내일 바뀌는 수준 |
| 문서화 | 최소한의 문서, 그러나 존재는 함 | 문서? 그런 건 나중에… |
| 피드백 | 실제 사용자 피드백 중심 | 상사 피드백 중심 (주로 “왜 안 나왔어?”) |
| 현실적 결과 | 품질 유지 + 점진적 개선 | “일단 내놨는데 고칠 게 많음” |

---

### 😅 즉, 판교식 “애자일하게”
- “우린 유연해야 해” → 사실상 **계획 없음**
- “빨리 MVP 만들어야 해” → **프로토타입 수준**으로 출시
- “유저 피드백 보면서 개선하자” → **유저가 QA 해주는 셈**
- “개발 일정은 유동적이야” → **일정이 계속 미뤄짐**

---

### 💬 그래서 업계에서는 이런 농담도 많아요
- “애자일하게 가자” = **“기획서 없이 개발하자”**  
- “스프린트 회의하자” = **“오늘 뭘 할지 즉석에서 정하자”**  
- “지속적 통합(CI)” = **“어제 코드가 오늘 깨진다”**  
- “MVP(최소 기능 제품)” = **“이 정도면 돌아가긴 하잖아”**

---

### 🤔 결론적으로
판교식 “애자일”은 **진짜 애자일이 아니라**,  
그저 **Ad-hoc 개발에 ‘유연함’이라는 포장지를 씌운 형태**예요.

즉,
> “계획 없음 + 빠른 땜빵 + 사후 수습 = 애자일이라 부름”

그걸 생략하고 “빨리빨리 돌려”만 남은 게 현실이에요.

## 🔹 기본 개발 단계(공통)
대부분의 방법론에서 기본적으로 거치는 단계는 대략 이렇게 나눌 수 있어요:  

1. **요구사항 분석(Requirements)**  
   - “무엇을 만들지?” 정의  
2. **설계(Design)**  
   - 시스템 구조, UI, 데이터 설계 등  
3. **구현(Implementation / Coding)**  
   - 실제 코드 작성  
4. **테스트(Testing / Verification)**  
   - 버그 수정, 기능 검증  
5. **배포/유지보수(Deployment / Maintenance)**  
   - 사용자에게 제공, 업데이트  

---

## 🔹 방법론별 특징
| 방법론 | 단계 순서 | 반복/건너뛰기 특징 | 느낌 |
|--------|------------|------------------|------|
| **폭포수(Waterfall)** | 1 → 2 → 3 → 4 → 5 | 반복 없음, 한 단계 끝나야 다음 단계 | ‘한 번에 쭉’ |
| **V-모델** | 폭포수 + 테스트 단계 매칭 | 반복 없음, 설계 단계별 대응 테스트 | 설계 중심 품질 |
| **나선형(Spiral)** | 1→2→3→4 반복 | **반복적**, 위험 관리 중심 | 점점 개선 |
| **프로토타이핑** | 1→3→4 반복 (설계 생략 가능) | **반복적**, 설계 없이 구현 먼저 | ‘돌려보면서 만든다’ |
| **RAD** | 1→3→4 반복, 2(설계)는 최소화 | 반복적, 빠른 개발 중심 | 프로토타입+빠른 피드백 |
| **애자일(Scrum, XP)** | 1→2→3→4 반복, 짧은 주기 | 단계 간 유연, 설계/코딩/테스트 혼합 | 지속적 개선 |
| **Ad-hoc / 카우보이 코딩** | 3→4 임의 반복 | 설계/분석 건너뛰기 | 즉흥적, 무계획 |

---

### 🔹 핵심 이해 포인트
1. **폭포수형** = 순차적, 단계마다 완료 후 진행 → 변경 힘듦  
2. **반복형/프로토타입형** = 구현 후 테스트→수정 반복 → 유연  
3. **애자일형** = 짧은 주기로 설계·구현·테스트 → 계속 개선  
4. **무계획형** = 설계 거의 없이 구현→테스트 반복 → 리스크 큼  

---

즉, **“단계를 반복하는지, 건너뛰는지, 순서를 꼭 지키는지”**가 방법론 차이의 핵심이에요.  
판교식 “애자일”은 사실상 **순서 무시 + 반복만 존재 + 설계 생략** 형태라 보면 됩니다.

## 🌀 애자일의 하위 개념 & 관련 방법론 요약

---

### 1. **스크럼 (Scrum)**
> “애자일의 가장 유명한 실천 방식”

#### 🔸 기본 개념
- 프로젝트를 **짧은 주기(스프린트)** 로 쪼개서 진행 (보통 2~4주)
- 각 스프린트마다 “작동 가능한 제품 조각(Increment)”을 만듦

#### 🔸 핵심 구성
| 개념 | 설명 |
|------|------|
| **스프린트 (Sprint)** | 일정 기간 동안 목표 기능을 완성하는 단위 사이클 |
| **스크럼 마스터** | 팀이 스크럼 원칙을 지키도록 돕는 관리자 (지시보단 조정) |
| **프로덕트 오너 (Product Owner)** | 요구사항 우선순위 결정, 제품 방향 제시 |
| **개발팀 (Developers)** | 실제 기능을 만드는 사람들 |
| **백로그 (Backlog)** | 해야 할 일 목록 (기능, 버그, 개선 등) |
| **스탠드업 미팅 (Daily Scrum)** | 매일 짧게 “어제 뭐 했고, 오늘 뭐 할지, 문제는 뭔지” 공유 |
| **리뷰 & 회고 (Review / Retrospective)** | 스프린트 끝나고 결과 확인, 다음 개선점 논의 |

#### 🔸 요약하면
> “짧은 주기로 계획 → 개발 → 리뷰 → 개선을 반복하는 팀 자율 중심 방식”

---

### 2. **칸반 (Kanban)**
> “작업 흐름을 시각화하는 방식”  

#### 🔸 특징
- **보드 기반 관리**: To Do → Doing → Done  
- 작업 상태를 시각적으로 관리  
- 스프린트 없이 **지속적인 흐름(Flow)** 강조  
- 생산 라인처럼 “현재 처리 중인 일(WIP)” 수를 제한해서 과부하 방지  

#### 🔸 요약
> “스프린트 개념 없는 스크럼처럼, 유연한 태스크 관리 중심 방식”

---

### 3. **XP (Extreme Programming)**
> “애자일 중에서도 개발 품질에 미친 놈(?)”

#### 🔸 핵심 실천법
| 개념 | 설명 |
|------|------|
| **TDD (Test Driven Development)** | 테스트를 먼저 만들고, 그걸 통과시키는 코드를 짬 |
| **Pair Programming** | 두 개발자가 한 자리에서 같이 코딩 |
| **Continuous Integration (CI)** | 코드가 합쳐질 때마다 자동 빌드/테스트 |
| **Refactoring** | 코드 기능은 그대로, 구조만 개선 |
| **Simple Design** | 복잡한 설계보단 단순함 유지 |

#### 🔸 요약
> “테스트, 코드 품질, 팀 협업을 극단적으로 밀어붙이는 방식”  

---

### 4. **Lean (린 개발)**
> “제조업의 낭비 제거 철학을 소프트웨어에 적용”

#### 🔸 주요 원칙
- 낭비 제거 (불필요한 문서, 미사용 기능 등)
- 가능한 빨리 사용자에게 전달
- 의사결정은 늦게 하되, 정보는 빠르게 얻기
- 팀의 자율성과 책임 강화  

#### 🔸 요약
> “최소한의 자원으로 최대 가치 창출”

---

### 5. **DevOps**
> “개발(Dev) + 운영(Ops) 융합 철학”

#### 🔸 핵심 목표
- **자동화, 협업, 지속적 배포**
- 코드가 커밋되면 → 자동 빌드 → 테스트 → 배포까지 이어짐 (CI/CD)
- 개발과 운영의 벽을 없애서 빠른 피드백  

#### 🔸 요약
> “개발 끝나면 끝이 아니라, 배포·운영까지 자동화된 연속 프로세스”

---

### 6. **MVP (Minimum Viable Product)**
> “최소 기능 제품” — 완벽보다 빠른 실험

#### 🔸 예시
- “모든 기능 완성은 나중에, 핵심 기능만 돌아가게 만들어서 시장 반응 보자.”
- 실패 비용을 줄이고, 진짜 필요한 기능을 검증하는 방식.  

---

## 💡 정리 이미지 (비유적으로)
| 방법론 | 초점 | 개발자 체감 |
|---------|--------|-------------|
| **Scrum** | 팀 주기적 반복, 협업 | "우리 이번 2주에 뭐 만들까?" |
| **Kanban** | 시각화된 업무 흐름 | "보드에 있는 거부터 하나씩 해보자" |
| **XP** | 품질과 테스트 중심 | "테스트 먼저 짜고, 코드 리팩터링하자" |
| **Lean** | 낭비 최소화 | "불필요한 회의나 문서 줄이자" |
| **DevOps** | 개발→배포 자동화 | "커밋하면 바로 배포되게 해놨다" |
| **MVP** | 빠른 실험 | "일단 핵심 기능만 만들어보자" |

---

요약하자면:  
> **스크럼은 애자일의 실전형 틀**,  
> **XP는 개발 품질 강화형**,  
> **칸반은 시각적 유연형**,  
> **린은 효율 철학**,  
> **DevOps는 기술적 자동화**,  
> **MVP는 실험적 시장 테스트**
