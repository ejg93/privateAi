이건 **TypeScript의 “유니온 타입 (Union Type)”** 문법입니다.

```ts
type Role = 'user' | 'assistant'
```

이 의미를 풀면:

- `Role`이라는 **타입 별칭(type alias)**을 만들고
- 값은 **'user' 또는 'assistant' 둘 중 하나만 허용**한다는 뜻이에요

즉:

```ts
let r: Role

r = 'user'       // ✅ 가능
r = 'assistant'  // ✅ 가능
r = 'admin'      // ❌ 에러
```

---

### 📌 이걸 뭐라고 부르냐면

좀 더 정확히는:

- **String Literal Type + Union Type**

구성은 이렇게 되어 있어요:

- `'user'`, `'assistant'` → **문자열 리터럴 타입**
- `|` → **유니온 (OR)**

---

### 📌 왜 쓰냐?

이걸 쓰는 이유는 간단해요:

👉 **허용 가능한 값만 강제해서 실수를 막기 위해**

예를 들어:

```ts
function sendMessage(role: Role) {
  // ...
}
```

이렇게 하면:

```ts
sendMessage('user')     // OK
sendMessage('assistant')// OK
sendMessage('system')   // ❌ 컴파일 에러
```

---

### 📌 TSX에서 쓰는 이유

React (TSX)에서는 이런 식으로 많이 써요:

```tsx
type Props = {
  role: Role
}
```

👉 컴포넌트에 들어오는 값이 정확히 제한됨

---

> 문자열들에서 따올 거니까 as const 써야 각각 값으로 인식된다

👉 **거의 맞는데, 정확히는 이렇게 말하는 게 좋음:**

👉 `as const`는  
**값을 "그 값 그대로" 타입으로 고정시킴**

---

## 비교 보면 바로 이해됨

### ❌ as const 없음

```ts
const arr = ['user', 'assistant']
```

👉 타입:

```ts
string[]
```

👉 `'user'` → 그냥 string으로 퍼짐

---

### ✅ as const 있음

```ts
const arr = ['user', 'assistant'] as const
```

👉 타입:

```ts
readonly ['user', 'assistant']
```

👉 `'user'` → `'user'` 그대로 유지

---

✔ 그래서 네 말 정리하면:

👉 **"각 값을 독립적인 타입으로 유지하려고 as const를 쓴다"**  
→ 이게 정확한 표현

# 🔹 2. typeof — 여기서 진짜 중요

너가 말한:

> "변수형 가져오는거"

👉 맞긴 한데, JS랑 TS에서 의미가 다름

---

## ✅ JS에서 typeof

```js
typeof 'user' // "string"
```

👉 **값 → 타입 문자열 반환 (런타임)**

---

## ✅ TS에서 typeof (지금 우리가 쓰는 거)

```ts
const arr = ['user', 'assistant'] as const

type T = typeof arr
```

👉 **"값 → 그 값의 타입을 가져온다" (컴파일 타임)**

---

## 🔥 핵심 차이

| 구분 | JS typeof | TS typeof |
|------|----------|-----------|
| 시점 | 실행 중 | 컴파일 |
| 결과 | "string" | 타입 자체 |
| 용도 | 디버깅 | 타입 추출 |

---

# 🔹 3. 전체 흐름 다시 정리

이 한 줄이 핵심임:

```ts
const STATUS = ['idle', 'loading'] as const
type Status = typeof STATUS[number]
```

---

## 단계별로 보면

### 1️⃣ 값 정의

```ts
['idle', 'loading']
```

---

### 2️⃣ as const

👉 값을 타입으로 고정

```ts
readonly ['idle', 'loading']
```

---

### 3️⃣ typeof

👉 그 "값의 타입" 가져옴

---

### 4️⃣ [number]

👉 배열 안에 있는 값들 꺼냄

---

## 최종 결과

```ts
'idle' | 'loading'
```

# 🔹 1. “리터럴 타입 정의” — 맞냐?

> 어떤 변수의 타입을 명확하게 정해준다

👉 ❌ **반은 맞고, 반은 틀림**

좀 더 정확하게 말하면:

👉 **“값 하나를 타입으로 만든 것”**

---

## 예시로 보면

```ts
let a: string = 'user'
```

👉 `a`는 아무 문자열 가능

---

```ts
let a: 'user' = 'user'
```

👉 이건 완전 다름

- `'user'` → **값이면서 동시에 타입**
- `'user'` 말고는 절대 안됨

---

## ✔ 정확한 정의

👉 **리터럴 타입 = 특정 값 하나만 허용하는 타입**

---

# 🔹 2. 핵심 질문:  
## ❗ 왜 “값을 타입으로 써야 하냐?”

이거 진짜 중요한 질문인데, 답은 하나임:

👉 **"실수를 막기 위해" + "의도를 코드에 강제하기 위해"**

---

## 🔥 문제 상황 (JS 스타일)

```ts
function send(role: string) {}
```

```ts
send('user')
send('assistant')
send('usr')  // ❌ 오타인데 통과됨
```

👉 JS/TS 기본 string은 너무 넓음

---

## 🔥 해결 (리터럴 타입)

```ts
type Role = 'user' | 'assistant'
```

```ts
function send(role: Role) {}
```

```ts
send('user')      // OK
send('assistant') // OK
send('usr')       // ❌ 바로 에러
```

---

👉 즉:

> 문자열을 쓰는 게 아니라  
> **"허용된 문자열만 쓰게 강제"하는 것**

---

# 🔹 3. 이 코드 해석해보자

```ts
type Role = 'user' | 'assistant'

type Message = {
  role: Role
  content: string
}
```

이건 이렇게 읽으면 됨 👇

---

## 💬 사람 말로 번역

👉 Message 객체는:

- `role`은 반드시 `'user'` 또는 `'assistant'`
- `content`는 아무 문자열

---

## 실제 사용

```ts
const msg: Message = {
  role: 'user',
  content: 'hello'
}
```

```ts
const msg: Message = {
  role: 'admin',  // ❌ 에러
  content: 'hello'
}
```
