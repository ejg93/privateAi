- **128코어(CPU 128개)**

### ⚙️ 2. “부하 16배를 버티고 있다”

즉,  
- “DB가 가진 총 데이터 용량”  
- “서버가 처리할 수 있는 트래픽 양(Request per second)”  
- “RAM 용량(1TB)”  

이 셋은 **완전히 다른 차원**의 개념이에요.

| **RAM 1TB** | **책상 크기** | 한 번에 꺼내서 펼쳐볼 수 있는 데이터 양이에요. (캐시, 쿼리 결과 등) |
| **Request per second (RPS)** | **책을 빌리는 손님 수/초** | 서버가 초당 처리할 수 있는 요청량이에요. |

CPU(=사서 수, 128명)가 많으면 손님(요청)을 동시에 더 많이 처리할 수 있는 거예요.

## 💾 그러면 “1TB DB”의 실제 의미는?
- DB 서버가 가진 **저장 데이터 용량**이에요. (디스크 기준)
- 즉 “전체 테이블, 인덱스, 로그 파일 합쳐서 1TB짜리 DB다”라는 뜻.
- RAM 1TB는 그중 일부를 캐싱해서 빠르게 접근하는 용도.

그래서 보통 이런 관계가 됩니다 👇

| 구분 | 설명 |
|------|------|
| **디스크(SSD/HDD)** | 전체 데이터 저장소 (DB 사이즈 = 1TB) |
| **메모리(RAM)** | 최근 자주 쓰이는 데이터 캐시, 버퍼, 인덱스 등 저장 |
| **CPU(128코어)** | 동시에 요청 처리, 쿼리 실행 등 담당 |
| **RPS** | 클라이언트 요청량 (CPU·RAM이 충분할수록 높음) |

# 추정 예시 (RDBMS)

1. **초간단 인덱스 PK 조회 (메모리에서 전부 처리, 거의 I/O 없음)**  
- per-core 처리량 가정: **1,000 RPS** (보수적) → 전체 현재 = `128 × 1,000 = 128,000 RPS` → 평소 = `128,000 ÷ 16 = 8,000 RPS`  
- per-core 공격적 가정: **2,000 RPS** → 현재 = **256,000 RPS** → 평소 = **16,000 RPS**

2. **일반 OLTP(읽기 우세, 일부 쓰기, 단순 쿼리)**  
- per-core 200 RPS → 현재 = **25,600 RPS** → 평소 = **1,600 RPS**

3. **혼합 트랜잭션(쓰기 많음, 커밋·fsync 필요)**  
- per-core 10–50 RPS → 현재 = **1,280 ~ 6,400 RPS** → 평소 = **80 ~ 400 RPS**  
(현실적으로 쓰기 중심이면 디스크 fsync·transaction log가 병목이 되어 RPS가 크게 떨어짐)

4. **복잡한 쿼리(조인/집계/OLAP 스타일)**  
- per-core 1–5 RPS → 현재 = **128 ~ 640 RPS** → 평소 = **8 ~ 40 RPS**

5. **메모리/세션 기반(동시 세션 수로 제한)**  
- 1TB RAM으로 세션당 10MB 가정 → 동시 약 `~104,800 세션` (이론) → 평소는 그 ÷16 → **~6.5k 세션**  
(실제로는 메모리 외에도 connection pool, OS limits, DB 내부 구조가 제약)

# 계산 공식을 한 줄로
- **총 이론 RPS ≈ 128(코어) × per-core RPS**  
- **평소(정상) = 총 이론 RPS ÷ 16**  
- 또는 **RPS = 동시(Concurrency) / 평균응답시간(초)** — 모니터링 수치로 직접 계산 가능.

예: 평균 쿼리 응답시간이 10ms(0.01s)이고 동시 쿼리(concurrency)가 1,280이면 RPS = 1,280 / 0.01 = 128,000 RPS.

---

# 중요한 실무 포인트 (왜 범위가 넓은지)
- **단순 조회**는 CPU·메모리로 거의 해결 → 매우 높은 RPS 가능.  
- **쓰기/트랜잭션**은 디스크 동기화(I/O)에 민감 → RPS 급감.  
- **쿼리 단일 스레드 성능**: 복잡한 쿼리는 단일 코어에서 오래 잡아먹기 때문에 코어 수가 많아도 한 쿼리당 속도는 느릴 수 있음.  
- **캐시 히트율**(버퍼풀/OS cache)이 높으면 훨씬 많은 RPS 처리 가능.  
- **네트워크, connection pool, DB 설정(max_connections, work_mem 등)** 도 병목.

## 🧮 1. RPS(Requests Per Second)는 “요청 수”, **“요청자(사용자 수)”가 아님**
- **RPS = 초당 쿼리 실행 횟수**예요.  
  → 한 사용자가 한 초에 10번 요청하면, 그건 10 RPS로 계산됩니다.  
  → 따라서 **몇 명이 요청하는지(동시 사용자 수)**는 따로 알 수 없어요.  
- 사용자 수를 알고 싶다면,  
  → “동시 접속 세션 수”  
  → “Connection Pool 수”  
  → “Active session/connections”  
  등을 봐야 합니다.

> 예시:  
> - 100명이 각각 초당 2개의 쿼리를 날리면 → 200 RPS  
> - 10명이 초당 20개 쿼리를 날려도 → 역시 200 RPS  
>   
> 💡 즉, RPS는 단순히 **‘서버에 날아오는 쿼리의 빈도’**입니다.

맞아요.  
`SELECT`, `INSERT`, `UPDATE`, `DELETE`가 같아 보이지만  
→ 실제로는 **서버 리소스 사용 프로파일이 전혀 달라요.**

아래는 일반적인 RDBMS(예: MySQL, PostgreSQL, Oracle 등) 기준입니다 👇  

| 쿼리 유형 | CPU 부하 | 메모리 부하 | 디스크 부하 | 특징 |
|------------|-----------|-------------|--------------|------|
| **SELECT (간단한 PK 조회)** | 낮음 | 낮음 | 거의 없음 (캐시 적중 시) | 버퍼 캐시 덕분에 CPU 위주, 매우 빠름 |
| **SELECT (조인/집계/서브쿼리 등 복잡)** | 높음 | 중간~높음 | 높음 (임시 테이블, 정렬, 스캔 발생) | CPU, I/O 둘 다 많이 씀 |
| **INSERT (단건)** | 낮음 | 낮음 | 중간 (로그 기록 필요) | InnoDB 로그 I/O 발생 |
| **INSERT (벌크)** | 중간~높음 | 높음 (버퍼/정렬/트랜잭션 캐시) | 높음 | 대량 쓰기 시 디스크 I/O 급상승 |
| **UPDATE** | 중간 | 중간 | 높음 (redo log + row rewrite) | 변경된 레코드 + 인덱스 갱신 비용 |
| **DELETE** | 중간 | 낮음 | 높음 (트랜잭션 로그 + 인덱스 갱신) | 대량 삭제 시 CPU도 올라감 |
| **TRUNCATE / DROP / ALTER** | 낮음~중간 | 낮음 | 매우 높음 (메타데이터 조작) | 스키마 락, 전체 I/O 발생 가능 |
| **SELECT ... JOIN ... GROUP BY ... ORDER BY** | 높음 | 높음 | 높음 | CPU+RAM+디스크 풀 부하 |

---

## 💾 3. “대량 쿼리”의 경우, 코어/메모리 부하가 폭발적으로 다름

- **대량 SELECT** → CPU 연산 + 정렬/집계 버퍼 때문에 RAM 사용 증가  
- **대량 INSERT/UPDATE** → WAL(Write Ahead Log)나 redo log가 병목, I/O 중심  
- **대량 DELETE** → 트랜잭션 로그 + 인덱스 업데이트로 I/O + CPU 둘 다 부하  
- **BULK LOAD (대용량 import)** → 병렬 처리 효율이 좋으면 코어가 꽉 찰 수 있음  
- **복잡한 조인** → 메모리에서 못 끝나면 디스크 temp table로 spill 발생 → 속도 급감

결국,
> “128코어, 1TB RAM이니까 무조건 빠르다”  
> 는 말은 **‘쿼리가 가벼울 때’만** 성립합니다.

---

## 🧠 4. 그래서 실무에서는 이렇게 구분해요

| 구분 | 주된 리소스 | 예시 |
|------|--------------|------|
| **CPU-bound 쿼리** | 계산량이 많음 | JOIN, GROUP BY, JSON 연산, 계산식 |
| **I/O-bound 쿼리** | 디스크 접근 많음 | 풀스캔, 로그 쓰기, 대량 INSERT/DELETE |
| **Memory-bound 쿼리** | 임시 테이블/정렬 많음 | ORDER BY, DISTINCT, WINDOW 함수 |
| **Network-bound** | 결과가 너무 큼 | 대량 resultset 전송 |

## 💡 5. 실제 튜닝 시엔 이런 지표를 같이 봐야 정확히 판단 가능
- **avg query latency (ms)** — 쿼리 응답 시간  
- **Buffer pool hit ratio** — 캐시 적중률  
- **CPU utilization per core**  
- **IOPS / fsync latency** — 디스크 쓰기 속도  
- **Active sessions / waiting sessions**  
- **Query plan (EXPLAIN)** — CPU vs I/O 부하 파악

| 대량 쿼리 | 병목이 어디에 생기느냐에 따라 코어/램 사용이 달라짐 |

# 🧠 1. 과부하를 숫자로 보는 핵심 3축

## ① **Latency (응답시간)** ← 제일 중요
- avg 말고 **p95 / p99** 봐야 함
- 정상: p95 10~20ms  
- 과부하: p95가 **몇 배 튀거나 초 단위로 증가**

👉 의미:
> “요청 하나 처리하는 데 드는 공수가 늘어나고 있다”

---

## ② **리소스 사용률 (CPU / RAM / IO)**

### CPU
- 70~80%: 정상
- 90% 이상 지속: 위험
- 100% 고정: **CPU 과부하 확정**

---

### 메모리
- RAM 꽉 차고 → swap 발생하면 **사망 직전**
- DB라면:
  - buffer cache miss 증가 = 성능 급락

---

### 디스크 (이게 진짜 많이 놓침)
- **IOPS / await (대기시간)** 중요
- await 1ms → 정상  
- await 10~50ms → 위험  
- await 100ms+ → 이미 병목

---

## ③ **Queue / 대기열 / 동시성**

이게 핵심입니다.

- DB connection이 쌓인다
- lock wait 증가
- thread pool 대기 증가

👉 이건 의미가 뭐냐면:

> **“요청이 처리 못 되고 줄 서고 있다” = 진짜 과부하**

---

# 🔥 2. 과부하를 한 줄로 정의하면

그래서 발생하는 현상:

- latency 증가
- queue 증가
- timeout 발생
- 에러 증가
