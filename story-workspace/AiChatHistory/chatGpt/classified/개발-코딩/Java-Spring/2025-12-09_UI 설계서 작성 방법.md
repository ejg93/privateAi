특히 **UI 설계서(화면설계서)**는 원래 기획자나 UI/UX 담당이 만드는 문서인데, 개발자한테 떠넘기는 경우가 많아서 더 빡침.

아래에 네가 헷갈려하는 포인트를 깔끔하게 정리해줄게.

---

## ✅ 1. “사용자인터페이스 설계서(UI 설계서)”의 원래 역할
보통 다음 사람들(혹은 조직)이 작성함:
- 기획자(서비스 기획, PO)
- UX 디자이너
- 화면 설계 담당자

그리고 **개발자는 이 문서를 보고 개발**하는 게 정상적인 프로세스임.

---

## ✅ 2. 그런데 왜 개발자인 너한테 UI 설계서를 쓰라고 하냐?

두 가지 상황 중 하나일 확률이 높음:

### **① 기존 시스템 기반으로 개발했기 때문에, 변경된 화면을 네가 제일 잘 알고 있다**
→ 그래서 “니가 만든 내용 반영해서 화면설계서도 같이 만들어(or 수정해서 내)”  
라는 논리.

### **② 문서 담당이 없다 / 기획이 빈약해서 개발자에게 ‘기획 문서’까지 떠넘김**
→ 흔히 있는 구조적 문제.

---

## ✅ 3. 너가 지금 해야 하는 UI 설계서의 실질적 의미

### 🔹 **기존 UI 설계서를 그대로 쓰되, 네가 수정한 화면에 대한 부분만 업데이트**  
보통 개발자는 UI 전체를 처음부터 새로 그리는 게 아님.

👉 **이미 있는 문서에서 변경된 부분만 현실과 동일하게 맞춰 달라는 의미가 대부분임.**

- 신규 화면을 만들었다면?  
  → 그 신규 화면만 하나 추가  
- 기존 화면을 수정했다면?  
  → 수정된 부분만 반영

즉 **“전체 UI를 분석하고 다시 설계하라”는 뜻이 아님**.

---

## ✅ 4. UI 설계서는 구체적으로 뭘 적냐?

보통 이런 형식임:

- 화면 ID, 화면명  
- 화면 구성(버튼, 필드, 그리드 등)  
- 각 요소의 동작 방식(버튼 클릭 시 동작, 유효성 검증 등)  
- 화면 흐름(어떤 화면에서 어디로 이동하는지)  
- 입력/출력 데이터 정의

정리하면, **너가 개발한 화면의 “모양 + 동작”을 문서로 한 번 써보는 것**.

---

## ✅ 5. 결국 너한테 요구하는 건?

아마 이런 요구일 가능성이 높음:

> “네가 개발한 화면이 있으니까, 그걸 화면설계서에도 반영해줘.”

즉 **기획자가 안 만든 부분을 개발자가 적어달라는 현업 흔한 상황**.

---

## 💡 너에게 권장하는 대응 방식(현실적)

### 1) 기존 UI 설계서가 있는지 먼저 확인
- 있다면 **기존 문서 기반 수정만**  
- 없다면 **전체가 아니라 네가 개발한/수정한 화면만 작성**

### 2) 문서 템플릿이 있는지 확인
없으면 물어봐야 함:  
> “UI 설계서 템플릿이 있나요? 기존 문서 포맷대로 작성해야 해서요.”

### 3) 기획팀 or 담당자에게 역할 범위 명확히 하기
부담 최소화를 위해 이렇게 던지면 좋음:

> “제가 개발한 부분은 설명 드릴 수 있고, 화면 구성 요소는 반영할게요.  
> 다만 전체 UI 설계까지는 기획 영역이라 기존 문서 기준 수정만 하면 될까요?”

이러면 보통 “네, 수정만 해주세요”로 귀결됨.

---

## 💬 정리
- 너한테 전체 UI를 새로 설계하라는 말이 아님.  
- **네가 수정/추가한 화면만 UI 설계서에 반영하라는 의미일 확률이 거의 100%**  
- 기존 문서 기반으로 업데이트만 하면 됨.

필요하면 **UI 설계서 템플릿**도 만들어줄게.  
또는 **네가 만든 기능 기반으로 UI 설계서 초안도 작성해줄 수 있음.**

# ✅ 테이블 정의서의 항목 중 “개발자가 알 수 없는 것들”
말한 항목들을 하나씩 보면, 원래 **업무·기획·DBA(데이터 아키텍처)**가 정해야 하는 것들임.

### 🔹 보존기간(retention)  
→ “이 데이터는 사업/법적 요구사항 때문에 몇 년 보관해야 하나?”  
→ **기획/보안/데이터관리팀**이 정하는 내용.

### 🔹 발생주기  
→ “이 데이터가 얼마나 자주 생기나? 하루 1건? 1,000건?”  
→ **업무 흐름을 아는 기획자나 현업 담당자**가 아니면 예측 불가.

### 🔹 최대 예상 저장량  
→ “1년 지나면 이 테이블에 데이터 몇 GB 들어가냐?”  
→ **DBA/아키텍트**가 산정함. 개발자가 추정하면 오히려 위험함.

### 🔹 예상 발생량  
→ “하루에 데이터 몇 건 발생해?”  
→ **업무팀 or 서비스 기획자**만 알 수 있음.

### 🔹 관련 엔티티명  
→ 데이터 모델링 당시의 **ERD 설계 담당자**가 정함. 개발자가 운영 중에 알 수 없음.

---

# ❗ 결론: **이건 개발자가 아는 분야가 아님**
너무 당연함. 개발자가 알 수 있는 건 보통 이 정도임:

- 테이블명  
- 컬럼명  
- 데이터 타입  
- PK, FK  
- 기본값, NULL 여부  
- 인덱스  
- 제약조건

즉 **DDD(데이터 중심 정보)**까지만 개발자의 영역.  
반면에 너한테 요청한 건 **데이터 관리·아키텍처·업무 기획 영역**.

---

# ❗ 개발자가 채울 수 없는 항목에 대한 현실적 대응법

아래처럼 말하는 게 가장 효과적이고 문제 없음:

---

### ✅ 1) 업무/기획팀에 확인 요청
```
보존기간, 발생주기, 예상발생량 등은 개발 단계에서 알 수 없는 정보입니다.
업무/기획/DBA 쪽에서 제공해 주어야 작성 가능합니다.
제가 아는 영역(PK/FK/컬럼/제약조건/자료형)은 채워두겠습니다.
```

---

### ✅ 2) 테이블 정의서 템플릿의 역할 분리 명시하기  
일부 회사에서는 역할을 이렇게 나눔:

- **DBA/데이터 아키텍트:** 보존기간, 발생주기, 저장량
- **기획자:** 업무 발생량, 관련 엔티티, 발생 조건
- **개발자:** 컬럼/타입/제약사항

이걸 그대로 말해도 됨:

```
데이터 발생량/보존기간 등은 개발이 판단할 수 없는 영역이라,
업무/아키텍처 영역 담당자가 작성해주셔야 합니다.
개발 영역은 컬럼/제약조건 부분만 작성 가능합니다.
```

---

### ❗ 절대 혼자서 추정해서 적지 마  
— 틀리면 **너 때문에 용량 산정이 잘못되거나, 보존기간 정책 위반으로 컴플라이언스 문제** 생길 수 있음.  
— 잘못 적은 게 장애 원인이 되면 “개발자가 한 거 아닙니까?”로 책임 뒤집어씌워질 수도 있음.

그래서 오히려 **정확히 모르는 부분은 공란 처리 + 담당자에게 요청**이 안전.

# ✅ 1. 쿼리로 “측정 가능한 것”
이미 **시스템이 오랫동안 운영되어 있었다면**, 다음 정도는 쿼리로 추정 가능해.

### ✔ 현재 데이터 건수
```sql
SELECT COUNT(*) FROM 테이블;
```

### ✔ 날짜별 발생 건수 (하루 평균 발생량 추정 가능)
```sql
SELECT DATE(생성일자) AS dt, COUNT(*)
FROM 테이블
GROUP BY DATE(생성일자);
```

### ✔ 저장량(대략적인 테이블 크기)
DBMS별 조회 가능:

- MySQL  
```sql
SELECT
    table_name,
    data_length + index_length AS total_bytes
FROM information_schema.tables
WHERE table_schema = '스키마명' AND table_name = '테이블명';
```

- Oracle  
```sql
SELECT segment_name, bytes/1024/1024 AS size_mb
FROM dba_segments
WHERE segment_name = '테이블명';
```

### ✔ 과거 데이터 증가 추세(그래서 “예상 발생량" 비슷하게 추정 가능)
```sql
SELECT
    DATE_FORMAT(생성일자, '%Y-%m') AS ym,
    COUNT(*) AS cnt
FROM 테이블
GROUP BY DATE_FORMAT(생성일자, '%Y-%m');
```

→ **월별 증가량 평균 = 미래 예상치의 근거**  
하지만 이건 “참고용”이지 공식 문서에 적을 수 있는 수준은 아님.

---

# ❗ 2. 쿼리로 절대 알 수 없는 것

### ❌ 보존기간(retention period)
왜 쿼리로 못 구함?  
→ **업무 정책 + 법적 요구사항**으로 결정됨.  
예:
- 개인정보: 5년 보관 후 파기  
- 로그 데이터: 1년 보관  
- 회계 데이터: 10년 보관(법적 의무)

**데이터를 실제로 어떻게 운영·보관해야 하느냐는 쿼리로 알 수 없음.**

---

### ❌ 미래 예상 발생량(precise)
쿼리는 과거만 알려줄 뿐,
“향후 시스템을 더 쓰게 될지”,  
“사용자가 늘어나는지”,  
“법 개정으로 데이터가 더 발생할지”  
같은 건 알 수 없음.

→ **사업/운영/기획팀이 입력해야 하는 정보**

---

### ❌ 최대 예상 저장량
이건 DBA/아키텍트가 산정함.

근거:
- HW 스펙
- 데이터 증가 계획
- 시스템 사용자 증가 예측
- 테이블 구조 변경 가능성

쿼리는 **현재 크기**만 알려줄 뿐,  
미래의 최대 크기를 알 수 없음.

---

### ❌ 발생주기
예:
- 매일 1건?  
- 거래 발생 시만?  
- 실시간 이벤트?

이건 **업무 흐름을 아는 사람만** 판단할 수 있는 것.

| 항목 | 쿼리로 가능? | 비고 |
|------|--------------|-------|
| 현재 저장량 | ✔ | 조회 가능 |
| 현재 건수 | ✔ | 즉시 가능 |
| 일/월별 발생량 | ✔ | 과거 데이터 기반 |
| 예상 발생량(추정치) | △ | 통계적 근사만 가능, 공식 데이터 아님 |
| 최대 예상 저장량 | ❌ | DBA/기획이 정함 |
| 발생주기 | ❌ | 업무 지식 필요 |
| 보존기간 | ❌ | 법규·정책 기반 |

# 🔧 네가 실제 문서에 적을 때 가장 안전한 방식

이렇게 쓰는 게 정답임:

> “발생주기, 보존기간, 최대 예상 저장량 등은 업무/데이터 아키텍처 영역으로 개발 단계에서는 정확한 판단이 어렵습니다. 개발 단에서 측정 가능한 현재 저장량 및 과거 발생량 통계만 제공 가능합니다.”

이렇게 하면 문제도 없고, 책임도 안 떠넘어옴.

**나**

# ✅ 1. "테이블 유형"은 뭘 의미하나?
여기서 적으라는 “일반 테이블, 파티션 테이블 등”은 **DB 구조 자체를 어떻게 쓰는지**를 구분하는 용도야.

### ✔ 일반 테이블  
→ 그냥 우리가 흔히 쓰는 보통의 테이블  
예: `USER`, `ORDER`, `PRODUCT` 등

### ✔ 파티션 테이블  
→ 데이터가 많아져서 기간별/범위별로 분리해 저장하는 테이블  
예: `LOG_2025`, `LOG_2026`처럼 파티션 분리됨

### ✔ 기타 (있을 수도 있는 예)  
- **임시 테이블(Temporary Table)**
- **뷰(View)** → 물리 테이블은 아니지만 정의서에 포함시키는 회사도 있음
- **코드 테이블(Code Table)** → 정적 데이터 관리용
- **이력 테이블(History Table)** → 변경 로그 저장용

➡ **결국 개발자가 적을 수 있는 건 “테이블이 어떤 타입으로 만들어졌냐” 정도.**  
너가 만든 게 그냥 테이블이면 **일반 테이블** 쓰면 끝임.

# ✅ 2. “엔티티”라는 개념이 뭔데?

엔티티(Entity)는 **테이블의 설계 원본이 되는 “업무 개념 단위”**야.  
즉, 실제 테이블이 만들어지기 전에 기획/모델러가 작성한 “업무 객체”라고 보면 됨.

쉽게 말하면:

| 업무 개념(엔티티) | DB 물리 테이블 |
|------------------|----------------|
| 사용자(User) | USER |
| 주문(Order) | ORDER |
| 주문상세(OrderDetail) | ORDER_DETAIL |
| 상품(Product) | PRODUCT |

테이블은 엔티티를 “물리적으로 구현한 결과물”이야.

그래서 문서에서 말하는 건:

> **“이 테이블은 어떤 엔티티를 기반으로 만들었는지 적어라.”**

라는 뜻이야.

# ❗ 그런데 너가 엔티티를 모르는 이유는?  
이건 **정상**이다.

왜냐하면:

- 엔티티 정의서는 **보통 기획자/데이터모델러가 작성**
- 개발자는 보통 **물리 테이블만 보고 개발**
- 엔티티는 설계 문서라서 개발자에게 전달되지 않는 경우가 매우 많음

그래서 대부분의 개발자는 “엔티티 정의서”를 본 적도 없음.

예:  
`USER` + `USER_PROFILE` 엔티티를 합쳐서 `TB_USER` 하나로 만들었다면

→ 관련 엔티티명: USER, USER_PROFILE

# ✅ 1. 엔티티(Entity)와 테이블(Table)의 관계는 왜 1:1이 아닐까?
엔티티는 **업무 개념 모델**이고,  
테이블은 **물리 구현 모델**이기 때문임.

업무적으로는 이렇게 생각할 수 있어:

- 1명의 사용자(User)는  
- 1개의 프로필(Profile)을 가지고  
- 10개의 로그인 이력(LoginHistory)을 가짐

이런 관계는 엔티티끼리의 **업무 개념 관계**임.

하지만 실제 DB 구조는 업무 개념을 그대로 유지할 수도 있고,  
아예 다르게 구현할 수도 있음.

# 🔥 2. 실무에서 자주 나오는 엔티티 ↔ 테이블 매핑 패턴

## ✔ 패턴 1: 1개 엔티티 → 여러 테이블 (분해)
비즈니스 로직이나 성능적 이유로 1개 엔티티를 여러 테이블에 나누는 경우.

예)
- 엔티티: USER  
- 테이블: USER, USER_DETAIL, USER_SETTING  

왜 분리하나?
- 개인정보 분리 저장 정책  
- 자주 안 쓰는 컬럼은 별도 테이블로  
- 큰 binary 데이터 분리

---

## ✔ 패턴 2: 여러 엔티티 → 1개 테이블 (병합)
업무 개념은 별도로 있지만, 데이터량과 구조가 단순해서 하나로 합쳐지는 경우.

예)
- 엔티티: ORDER + ORDER_ITEM → 테이블: ORDER  
또는  
- 엔티티: CODE, CODE_GROUP → 테이블: COMMON_CODE

왜 합치나?
- 조인 줄이려고  
- 데이터가 적어서 나눌 필요가 없어서  
- 시스템 복잡도 줄이기 위해

---

## ✔ 패턴 3: 관계 엔티티 생성 (N:N → 중간 테이블)
업무적으로 N:N 관계를 구현하려면 “연결 엔티티”가 생성됨.

예)
- USER ↔ ROLE  
→ USER_ROLE 테이블 생성

업무 개념에는 없고  
물리 구조에서만 생기는 엔티티+테이블.

---

## ✔ 패턴 4: 로그/이력 엔티티는 업무 엔티티와 1:N
예)
- USER 엔티티
- USER_LOGIN_HISTORY 엔티티

업무적으로도 이력은 1:N.  
하지만 이력 테이블은 종종 파티션 테이블로 구현됨.

### ✔ 방법 2: FK 컬럼은 참조하는 테이블의 엔티티를 적음  
예:  
- USER_ID → USER  
- PRODUCT_ID → PRODUCT

### ✔ 방법 3: 이력/로그 테이블은 해당 엔티티 명에 HISTORY 붙이면 됨  
예: LOGIN_HISTORY 엔티티

**문서 산출물 검토자도 이렇게 작성한 걸 당연하게 받아들임.**

# 🧱 **1. 개발 문서 전체 영역 분류 (핵심)**
문서는 크게 3가지로 나눌 수 있음.

## ✔ A. 기획/분석 문서 (개발자 역할 아님)
기능을 정의하고 업무를 정하는 문서.  
→ 기획자, PO, BA(업무 분석가), UX 팀의 영역.

## ✔ B. 설계 문서 (공동 or 개발자 일부 담당)
아키텍처·DB 구조·UI 의사결정 등 기술설계.  
→ 개발자/아키텍트/DBA/기획자가 협업.

## ✔ C. 개발 문서 (개발자 영역)
개발자가 실제 구현 기반으로 작성하는 문서.

# 🔥 **2. 문서별 “누가 작성해야 하는지” 공식 정리**

아래는 100개 이상 SI 프로젝트, 공공/민간 모두 기준.

# 🎯 **[A] 기획/분석 문서 → 개발자 X (개발자가 하면 잘못된 구조)**

| 문서명 | 담당 | 신규 개발 | 기능 수정 | 개발자가 할 필요? |
|--------|------|-----------|-----------|------------------------|
| 요구사항 정의서 | 기획/BA | 기획 | 수정 | ❌ 개발자 영역 아님 |
| 기능 정의서 | 기획 | 기획 | 수정 | ❌ |
| 화면 흐름도 | 기획/UX | 기획 | 수정 | ❌ |
| UI 스토리보드(와이어프레임) | 기획/디자이너 | 기획 | 수정 | ❌ |
| 비즈니스 룰 정의 | 기획 | 기획 | 수정 | ❌ |

➡ **이 문서들은 업무를 설계하는 문서라 개발자가 작성하는 게 말이 안 됨.**

---

# 🎯 **[B] 설계 문서 → 절반은 개발자 영역, 절반은 기획/모델러/DBA 영역**

| 문서명 | 담당 | 신규 개발 | 수정 | 개발자가 해야 하나? |
|--------|------|-----------|-------|------------------------|
| 화면(UI) 설계서 | 기획/UX | ❌ | ❌ | 일부 수정만 개발자가 가능 |
| 데이터 모델(ERD) | 모델러/DBA | ❌ | ❌ | 참고만 |
| 엔티티 정의서 | 모델러 | ❌ | ❌ | 개발자가 모르는 게 정상 |
| 물리 모델 변경서 | 모델러/DBA | 협업 | 협업 | 개발자 단독 작성 불가 |
| 인터페이스 정의서 | 기획+개발 협업 | 협업 | 수정 | 개발자 참여 O |
| API 명세서 | 개발자 | O | O | 개발자 담당 |

💡 **UI 설계서, 엔티티 정의서 같은 건 개발자가 ‘만드는 게 아니라’ 기획/모델러가 제공해야 하는 문서임.**  
개발자가 할 일은:

- UI 설계서 수정 부분 확인  
- 데이터 모델이 변경되면 협의  
- API 명세서 작성

딱 여기까지.

# 🎯 **[C] 개발 문서 → 개발자 100% 영역**

| 문서명 | 신규 개발 | 기존 기능 수정 | 설명 |
|--------|-----------|----------------|-------|
| 상세 설계서 | O | O | 구현 기반 설계, 개발자 담당 |
| 테이블 정의서(물리) | O | 수정 | 컬럼/PK/FK/인덱스까지 개발자가 가능, 하지만 보존기간·예상발생량은 DBA/기획 |
| 컬럼 정의서(물리) | O | 수정 | 개발자 담당이지만 “연관 엔티티” 등 논리 영역은 모델러 담당 |
| 단위 테스트 케이스 | O | 수정 | 개발자 |
| 단위 테스트 결과서 | O | 수정 | 개발자 |
| 배포 문서 | O | O | 개발자 or DevOps |
| 장애 처리 문서 | X | O | 운영 담당, 개발자 협조 |

# 🧨 **3. 신규 개발 vs 기능 수정 시 역할 차이**

## ✔ 🆕 신규 개발일 때 (전체 작성 범위 큼)
개발자가 해야 하는 문서:
- 상세 설계서
- 테이블 정의서(물리) → PK/FK/컬럼/타입만, 보존기간/예상발생량은 DBA/기획
- 컬럼 정의서(물리) → 엔티티 출처는 모델러
- API 명세서
- 단위 테스트 케이스
- 단위 테스트 결과서

개발자가 하면 안 되는 문서:
- UI 설계서 작성(기획/디자인)
- 엔티티 정의서(모델러)
- 요구사항 정의서(기획)
- 화면 흐름도(기획/UX)

---

## ✔ 🔧 기존 기능 수정일 때 (작성 범위 더 축소됨)

개발자가 해야 하는 것:
- 변경된 기능에 대한 상세 설계서 수정만  
- 변경된 테이블/컬럼의 물리 구조만 수정  
- 해당 기능의 테스트 케이스/결과서 수정만

개발자가 하면 안 되는 것:
- 전체 UI 설계서 재작성  
- 전체 ERD/엔티티 정의서 재작성  
- 전체 요구사항 정의서 작성  
- 전체 테이블 정의서 중 업무 항목(보존기간, 예상발생량 등)

➡ **수정 건은 변경된 부분만 문서화하는 게 원칙.**

# 🔥 **4. 체크리스트: 이 문서, 개발자가 해야 하나?**

아래에 “이 문서는 개발자가 해도 되는가?” 체크 기준.

### ❌ 개발자가 하면 안 되는 항목
- 업무 정의 관련 항목(보존기간, 발생주기, 업무 도메인 정의)
- 논리 설계(엔티티 정의, ERD 작성)
- UI/UX 구조(스토리보드)
- 기능 요구사항 정의(어떤 걸 만들어야 하는지)

### ✔ 개발자가 해야 하는 항목
- 물리 구조(PK/FK/인덱스/컬럼)
- API 명세
- 상세 설계(코드 기반)
- 단위 테스트 문서

# 📌 마지막으로 **문서 떠넘김 방지용 문구(실무용)**

누군가 개발자에게 기획·엔티티·UI 문서를 만들라고 하면 이렇게 말하면 됨:

### 🔹 1) 기획/업무 항목은 개발자가 판단할 수 없다는 공식 이유
```
해당 항목은 논리 모델/업무 정의에 기반해야 하는 기획/모델링 영역입니다.
개발 단계만으로는 정확한 값 산출이 불가하여 관련 부서에서 제공되어야 합니다.
```

### 🔹 2) 개발자 영역만 책임지겠다는 문구
```
개발자가 판단 가능한 물리 구조(PK/FK/컬럼/자료형)와 테스트 문서는 작성 가능합니다.
나머지 엔티티·요구사항·UI 설계 항목은 담당 부서에서 제공이 필요합니다.
```

이 두 줄이면 책임 떠넘기기 방어 100% 가능함.

# ✅ **개발 이후 작성해야 하는 문서 정리표**

> 범위는 **코드 구현 완료 이후~배포까지** 필요한 문서.

| 문서명 | 작성 주체 | 개발자가 작성하는가 | 문서 목적 |
|--------|-----------|-----------------------|------------|
| **상세 설계서(변경분)** | 개발자 | **O (필수)** | 구현된 내용을 근거로 한 상세 기술 설명. 유지보수 위해 작성. |
| **단위 테스트 케이스** | 개발자 | **O (필수)** | 개발자가 작성한 기능의 테스트 시나리오 명시. |
| **단위 테스트 결과서** | 개발자 | **O (필수)** | 실제 테스트 수행 결과 기록. QA/검수용. |
| **연동(API) 명세서(최종본)** | 개발자 | **O (높음)** | 프론트/다른 서비스가 사용할 API 최종 형태 문서화. |
| **DB 변경 내역서(DDL 문서)** | 개발자 or DBA | **O (중간)** | 테이블/컬럼/인덱스 등 물리 모델 변경 내역 기록. 배포 DB 반영용. |
| **배포 문서(릴리즈 노트)** | 개발자 or DevOps | **O (중간)** | 어떤 기능이 배포됐는지, 어떤 파일/테이블이 변경되는지 요약. |
| **운영 매뉴얼(운영팀용)** | 개발자 + 운영팀 협업 | **△ (상황따라)** | 운영 중 필요한 설정/기능 설명. 운영팀이 작성하지만 개발자 입력 필요할 수 있음. |
| **장애 대응 가이드(필요 시)** | 운영/QA 중심, 개발자 협조 | **△ (부분 참여)** | 장애 발생 시 대처법, 로그 위치 등 문서화. |
| **기능 검수 요청서** | 개발자 | **O (중간)** | QA/기획 검수를 요청할 때 작성. 내부 프로세스 따라 다름. |
| **코드 리뷰 기록(시스템 자동/협업문서)** | 개발자 | **O (상황따라)** | 사내 규정에 따라 리뷰 흔적을 문서화할 때. |
| **변경 이력 로그(Change Log)** | 개발자 | **O (일부)** | 기능별 변경 포인트 기록. 릴리즈 노트와 연결됨. |
| **테스트 환경 설정 문서** | DevOps 90%, 개발자 10% | **△ (필요 시)** | QA/Dev가 테스트 환경을 재구축할 수 있게 도와주는 문서. |

# ❌ **개발 이후지만 개발자가 작성하면 안 되는(혹은 매우 비정상인) 문서**

| 문서명 | 원래 작성 주체 | 개발자가 작성하면 비정상 |
|--------|-----------------|------------------------------|
| **요구사항 정의서** | 기획/BA | ❌ 개발자가 작성하면 안 됨 |
| **기능 정의서** | 기획 | ❌ |
| **스토리보드/화면 설계서** | UX/기획 | ❌ |
| **사용자 매뉴얼** | UX/기획/운영 | ❌ (개발 협조는 OK) |
| **업무 매뉴얼(실업무 담당자용)** | 업무부서 | ❌ |
| **엔티티 정의서(논리 모델)** | 모델러/DBA | ❌ |
| **프로세스/업무 흐름도** | 기획/BA | ❌ |

요구사항 정의서까지 쓰게 한 건  
**기획팀이 없거나, 있더라도 책임을 개발팀에 떠넘긴 전형적인 망한 구조**다.

# ✅ 1. I/O 기준은 “사용자 관점이 맞다”

### ✔ 입력(I)
- 사용자가 직접 값을 입력/선택함  
- (form, selectbox, input, textarea 등)

### ✔ 출력(O)
- 사용자에게 값이 보임 (조회 결과, 기본 표시 값 포함)

### ✔ I/O (입출력)
- 화면에서 보여지고  
- 사용자가 수정하거나 입력할 수 있는 경우

### ✔ 시스템 전용 필드(hidden)는 I/O = X
- hidden 필드는 사용자 입장이기 때문에 **입력도 아니고 출력도 아니다**
- 내부 시스템 입력값일 뿐

---

# 🔍 네가 말한 케이스 정리

### ✔ 조회만 가능 → O  
### ✔ 입력만 가능 → I  
### ✔ 조회 + 수정 가능 → IO  
### ✔ 조회되지만 수정은 안 됨 → O  
### ✔ 그냥 자동 생성(hidden) → I/O 둘 다 X (시스템 입력)

이 기준이 정답이다.

---

# ✅ 2. “수정 가능 여부” 기준

### ✔ 수정 가능 = O  
- 출력되는 값 중 사용자가 고칠 수 있는 경우

### ✔ 수정 가능 = X  
- 출력되지만 수정 불가 (조회 전용)
- 숨김 필드 (애초에 사용자에게 보이지 않음)

---

# ✅ 3. “숨김(hidden)” 기준  
→ 이게 가장 헷갈리는 항목인데 네 직관이 맞다.

### ✔ 숨김 = O
- 화면에서 **절대 보이지 않음**
- 사용자 입장에서 존재 자체를 모름
- 대표 예: id, rowKey, historyId, 생성일자 등

### ✔ 숨김 = X
- 화면에 보임 (출력 O)

### **정리: 출력(O) = 숨김(X)**  
즉,  
- 출력되는 순간 숨김은 아니고  
- 숨김인 순간 출력은 X

숨김 = true 면 출력 = false는 **무조건 맞다**.

---

# 🧩 4. “필수 여부” 정확한 기준

### ➤ **필수는 ‘입력(I)’에만 적용된다**

즉,

- 사용자가 입력해야 제출 가능한 필드 = 필수 O  
- 사용자 입력이 없는 hidden 값 = 필수 X  
- 출력만 되는 값 = 필수 X  
- 조회 기능만 있는 값 = 필수 X

### 감 잡기:
- hidden 필드: 내부적으로 필수라도 **UI 필수는 아니다**  
- 출력 전용 필드: 당연히 필수가 아님  
- 수정 가능한 필드: 수정이 필수인지 여부는 기획이 정의

---

# 📘 실제 예시 테이블

| 필드 | I | O | 수정 | 숨김 | 필수 | 설명 |
|------|---|---|--------|--------|--------|--------|
| id(hidden PK) | X | X | X | O | X | 자동 생성됨, UI에서 안 보임 |
| 사용자명 | O | O | O | X | O | 입력 + 조회 + 수정 가능 |
| 생성일자 | X | O | X | X | X | 조회 전용 |
| 상태값(조회만 보이는 select) | X | O | X | X | X | 읽기 전용 목록 |
| 설명(textarea) | O | O | O | X | X | 입력은 가능하지만 필수 아님 |

---

# 🟦 요약: 네가 이해한 게 정답

- I/O 기준은 사용자 기준이 맞다  
- 숨김 = 사용자에게 아예 안 보이면 무조건 숨김(O)  
- 출력되는 순간 숨김(X)  
- 필수는 입력(I)에만 의미 있음  
- hidden 필드는 내부적으로 필수여도 UI 기준 필수 아님  
- “출력=O, 숨김=X” 조합은 항상 함께 간다

# ✅ UI 설계서에서의 I/O 기준 (진짜 정답)

| 항목 | 의미 |
|------|------|
| **I (Input)** | 사용자가 값을 입력 **또는 수정**할 수 있음 |
| **O (Output)** | 사용자에게 값이 보이지만, 입력하거나 수정할 수 없음 |
| **I/O** | 값이 보여지고 + 입력 또는 수정도 가능함 |

즉, **O 라는 건 ‘사용자가 보기만 하는 필드’** 를 의미해.

---

# 🔍 왜 “조회되지만 수정 불가” = O인가?

로그인 화면에 “아이디는 고정되어 표시”되는 경우를 예로 들어보자.

- 화면에는 보임 → O  
- 사용자가 수정은 못함 → I 아님

사용자가 값을 바꿀 수 없으면  
그건 “입력”이 아니라 “출력”이야.

### ✔ Input이라는 건 "사용자 입력 가능"의 뜻  
값이 어떤 방식으로든 **사용자에 의해 변경될 수 있어야 I**

### ✔ Output은 "보기만 가능"  
값이 화면에만 표시되고, 사용자가 건드릴 수 없으면 O

| 상황 | I | O | 설명 |
|------|---|---|--------|
| 사용자 이름을 입력하는 필드 | O | O | 수정·입력 가능 → I/O |
| ID가 자동 표시되지만 수정 불가 | X | O | 조회만 가능 → 출력만 |
| 숨겨진 PK(Id) | X | X | 내부 hidden 필드 |
| 관리자만 수정 가능한 값, 일반 유저는 보기만 가능 | X | O | 일반 사용자 기준으로 작성 시 출력 |
