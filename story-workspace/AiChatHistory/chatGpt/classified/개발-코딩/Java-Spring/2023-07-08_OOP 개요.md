OOP는 "Object-Oriented Programming"의 약어로, 객체지향 프로그래밍을 의미합니다. 객체지향 프로그래밍은 소프트웨어를 설계하고 구현하는 방법 중 하나로, 현실 세계의 객체를 프로그래밍에 적용하여 문제를 해결하는 접근법입니다.

객체지향 프로그래밍은 프로그램을 작은 단위인 객체로 분할하고, 이러한 객체들 간의 상호작용을 통해 기능을 구현합니다. 각각의 객체는 상태와 행위를 가지며, 데이터와 메서드로 구성됩니다. 객체는 클래스라는 템플릿을 기반으로 생성되며, 클래스는 객체를 생성하기 위한 설계도 역할을 합니다.

OOP의 주요 개념으로는 캡슐화(encapsulation), 상속(inheritance), 다형성(polymorphism)이 있습니다. 캡슐화는 객체의 상태와 행위를 하나로 묶어 외부에서의 접근을 제어하는 것을 의미하며, 정보 은닉과 관련이 있습니다. 상속은 기존의 클래스를 확장하여 새로운 클래스를 생성하는 것을 말하며, 코드의 재사용성과 확장성을 높이는데 도움을 줍니다. 다형성은 동일한 이름의 메서드나 함수가 다른 객체에서 다른 동작을 할 수 있는 능력을 말하며, 코드의 유연성과 가독성을 향상시킵니다.

객체지향 프로그래밍은 복잡한 문제를 모듈화하여 해결할 수 있는 장점이 있으며, 코드의 재사용성, 유지보수성, 확장성 등을 향상시킬 수 있습니다. 많은 프로그래밍 언어들이 객체지향 프로그래밍을 지원하며, 대표적인 예로는 Java, C++, Python 등이 있습니다.

Abstract Class와 Interface는 둘 다 추상 메서드를 가질 수 있지만, 그들 간에는 몇 가지 중요한 차이점이 있습니다.

1. 구조적 차이:
   - Abstract Class: 추상 클래스는 일반적인 클래스와 유사한 형태로, 일반 메서드와 추상 메서드를 모두 가질 수 있습니다. 추상 메서드는 구현되지 않은 메서드로, 하위 클래스에서 반드시 구현해야 합니다.
   - Interface: 인터페이스는 메서드의 명세(specification)를 정의하는데 사용됩니다. 모든 메서드는 추상 메서드로 선언되며, 기본적으로 메서드의 본문을 포함하지 않습니다.

2. 다중 상속:
   - Abstract Class: 추상 클래스는 다른 클래스를 상속받을 수 있으며, 다중 상속이 가능합니다. 즉, 추상 클래스는 클래스 계층 구조에서 하나의 클래스에만 직접 상속될 수 있습니다.
   - Interface: 인터페이스는 다른 인터페이스를 상속받을 수 있으며, 다중 상속이 가능합니다. 클래스가 인터페이스를 구현할 때에는 여러 개의 인터페이스를 동시에 구현할 수 있습니다.

3. 상태와 구현:
   - Abstract Class: 추상 클래스는 일반적인 클래스이므로 상태(인스턴스 변수)와 구현된 메서드(일반 메서드)를 가질 수 있습니다. 하위 클래스에서 공통적인 상태와 메서드를 상속받아 사용할 수 있습니다.
   - Interface: 인터페이스는 메서드 명세만을 포함하며, 상태(인스턴스 변수)나 구현된 메서드를 가질 수 없습니다. 클래스가 인터페이스를 구현할 때, 인터페이스에서 정의한 메서드들을 모두 구현해야 합니다.

4. 사용 목적:
   - Abstract Class: 추상 클래스는 특정 클래스들 간에 공통된 동작을 추상화하여 상속을 통해 코드 재사용과 일반화를 도모하기 위해 사용됩니다. 또한, 상태와 구현을 포함할 수 있으므로, 클래스 간의 계층 구조를 정의할 때 유용합니다.
   - Interface: 인터페이스는 클래스들이 어떤 메서드들을 구현해야 하는지를 명시하기 위해 사용됩니다. 여러 클래스 간의 동작을 표준화하거나 다형성을 구현하는 데에 유용합니다.

요약하면, Abstract Class는 클래스 계층 구조를 정의하고 구현된 메서드와 상태를 가질 수 있으며, Interface는 메서드 명세만을 정의하고 다중 상속을 지원합니다. 두 가지는 서로 다른 상황에 적합하게 사용될 수 있습니다.

**ChatGPT**

추상 클래스와 인터페이스에 대해 간단한 예시를 들어보겠습니다.

예시 1: 추상 클래스(Abstract Class)
```java
abstract class Animal {
    String name;
    
    public Animal(String name) {
        this.name = name;
    }
    
    abstract void makeSound();
    
    void sleep() {
        System.out.println(name + " is sleeping.");
    }
}

class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }
    
    void makeSound() {
        System.out.println(name + " says woof!");
    }
}

class Cat extends Animal {
    public Cat(String name) {
        super(name);
    }
    
    void makeSound() {
        System.out.println(name + " says meow!");
    }
}

public class Main {
    public static void main(String[] args) {
        Animal dog = new Dog("Bobby");
        dog.makeSound();  // 출력: Bobby says woof!
        dog.sleep();      // 출력: Bobby is sleeping.
        
        Animal cat = new Cat("Kitty");
        cat.makeSound();  // 출력: Kitty says meow!
        cat.sleep();      // 출력: Kitty is sleeping.
    }
}
```
위의 예시에서 `Animal` 클래스는 추상 클래스로 정의되었습니다. `makeSound` 메서드는 추상 메서드로 선언되어 하위 클래스에서 반드시 구현되어야 합니다. `sleep` 메서드는 구현된 메서드로, 모든 동물이 공통적으로 사용할 수 있습니다. `Dog`와 `Cat` 클래스는 `Animal`을 상속받아 `makeSound` 메서드를 구현합니다.

예시 2: 인터페이스(Interface)
```java
interface Vehicle {
    void start();
    void stop();
}

class Car implements Vehicle {
    public void start() {
        System.out.println("Car starts.");
    }
    
    public void stop() {
        System.out.println("Car stops.");
    }
}

class Motorcycle implements Vehicle {
    public void start() {
        System.out.println("Motorcycle starts.");
    }
    
    public void stop() {
        System.out.println("Motorcycle stops.");
    }
}

public class Main {
    public static void main(String[] args) {
        Vehicle car = new Car();
        car.start();  // 출력: Car starts.
        car.stop();   // 출력: Car stops.
        
        Vehicle motorcycle = new Motorcycle();
        motorcycle.start();  // 출력: Motorcycle starts.
        motorcycle.stop();   // 출력: Motorcycle stops.
    }
}
```
위의 예시에서 `Vehicle` 인터페이스는 `start`와 `stop` 메서드를 선언합니다. 클래스가 이 인터페이스를 구현할 때에는 인터페이스에서 정의한 메서드들을 모두 구현해야 합니다. `Car`와 `Motorcycle` 클래스는 `Vehicle` 인터페이스를 구현하고, `start`와 `stop` 메서드를 구현하여 자신만의 동작을 정의합니다.

이는 간단한 예시이며, 실제로 추상 클래스와 인터페이스는 더 복잡하고 다양한 상황에서 사용될 수 있습니다.

네, 맞습니다. `extends`는 클래스 간의 상속을 표현하며, 하나의 클래스만 상속받을 수 있습니다. 이를 단일 상속(single inheritance)이라고 합니다. 클래스는 다른 하나의 클래스를 확장(상속)하여 해당 클래스의 특성과 동작을 상속받을 수 있습니다.

육지동물과 바다동물은 서로 다른 환경에서 살기 때문에 호흡 방식이 다릅니다.

육지동물의 호흡:
육지동물은 대부분의 경우 폐를 이용한 간이 호흡을 합니다. 즉, 공기를 흡입하여 폐로 옮겨서 거기에서 산소를 흡수하고 이산화탄소를 방출합니다. 육지동물은 기름진 폐를 가지고 있으며, 폐에는 폐포(alveoli)라고 불리는 작은 구조물이 존재하여 효과적인 가스 교환이 가능합니다. 주로 기관지와 기관지 세포로 이루어진 호흡기계를 통해 공기가 폐로 이동하며 호흡이 이루어집니다.

바다동물의 호흡:
바다동물은 주로 물 속에서 살기 때문에 물을 이용한 간이 호흡을 합니다. 이러한 종류의 호흡은 물 속에서 산소가 녹아있는 물을 직접 흡입하여 가스 교환이 이루어집니다. 이산화탄소는 그들의 몸에서 물로 방출되어 물로 이동하고, 반대로 산소는 물 속에서 그들의 몸으로 흡수됩니다. 바다동물은 지느러미, 활주로, 홍합 등의 기생으로 호흡하며, 이러한 기관을 통해 물 속에서 산소를 얻고 이산화탄소를 방출합니다.

육지동물과 바다동물은 서로 다른 환경에서 살기 때문에 호흡에 적합한 기구를 갖추고 있습니다. 육지동물은 공기로 호흡하고 바다동물은 물로 호흡하여 각각의 환경에서 생존하고 성장할 수 있습니다.

@xmlrootelement를 사용하는 예시

---

**ChatGPT**

`@XmlRootElement`는 Java 언어에서 JAXB(Java Architecture for XML Binding) 라이브러리를 사용할 때 사용되는 주석(annotation)입니다. 이 주석은 Java 객체를 XML로 변환하거나 XML을 Java 객체로 역직렬화할 때, XML의 루트 요소(root element)를 지정하는 데 사용됩니다.

아래는 `@XmlRootElement`를 사용하는 간단한 예시입니다.

```java
import javax.xml.bind.annotation.XmlRootElement;

@XmlRootElement
public class Student {
    private int id;
    private String name;
    private int age;
    
    public Student() {
        // 기본 생성자는 JAXB에서 필요합니다.
    }
    
    public Student(int id, String name, int age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }

    // 게터와 세터, toString 메서드 등은 생략합니다.
}
```

위의 예시에서 `Student` 클래스는 `@XmlRootElement` 주석으로 표시되어 있습니다. 이렇게 설정하면 JAXB는 해당 클래스가 XML로 직렬화될 때 XML의 루트 요소가 되도록 합니다. 즉, `Student` 객체가 XML로 변환될 때 XML의 최상위에 `<Student>` 요소로 감싸질 것입니다.

이제 `Student` 객체를 XML로 변환하고 역직렬화하는 예시를 보겠습니다.

```java
import javax.xml.bind.JAXB;
import java.io.StringReader;
import java.io.StringWriter;

public class Main {
    public static void main(String[] args) {
        // Java 객체를 XML로 변환
        Student student = new Student(123, "John", 25);
        StringWriter writer = new StringWriter();
        JAXB.marshal(student, writer);
        String xmlString = writer.toString();
        System.out.println(xmlString);
        
        // XML을 Java 객체로 역직렬화
        String xml = "<Student><id>456</id><name>Alice</name><age>22</age></Student>";
        Student deserializedStudent = JAXB.unmarshal(new StringReader(xml), Student.class);
        System.out.println(deserializedStudent);
    }
}
```

위의 코드에서 `JAXB.marshal` 메서드를 사용하여 `Student` 객체를 XML로 변환하고, `JAXB.unmarshal` 메서드를 사용하여 XML을 `Student` 객체로 역직렬화합니다. 이때, `Student` 클래스가 `@XmlRootElement`로 표시되어 있기 때문에 JAXB는 해당 클래스를 기준으로 XML과 Java 객체 간의 변환을 수행합니다.

실행 결과는 다음과 같을 것입니다.

XML 변환 결과:
```
<Student>
    <id>123</id>
    <name>John</name>
    <age>25</age>
</Student>
```

역직렬화 결과:
```
Student{id=456, name='Alice', age=22}
```

위와 같이 `@XmlRootElement`를 사용하면 JAXB가 Java 객체와 XML 사이의 매핑을 쉽게 처리할 수 있습니다.

**ChatGPT**

`@Data`, `@JsonInclude`, `@ToString`, `@Schema`는 Java 언어와 관련된 여러 가지 유용한 주석들입니다. 각각의 주석에 대해 간단히 설명하겠습니다.

1. `@Data`:
`@Data`는 Lombok 프로젝트에서 제공하는 주석으로, 클래스에 적용할 경우 해당 클래스에 대한 기본 메서드들을 자동으로 생성해 줍니다. 이 주석을 사용하면 자동으로 `equals`, `hashCode`, `toString`, `Getter`, `Setter` 등의 메서드들을 생성하여 코드를 간결하고 가독성 좋게 만들어 줍니다.

```java
import lombok.Data;

@Data
public class Person {
    private String name;
    private int age;
}

// @Data를 사용한 클래스는 자동으로 Getter, Setter, equals, hashCode, toString 등을 생성합니다.
```

2. `@JsonInclude(JsonInclude.Include.NON_NULL)`:
`@JsonInclude`는 Jackson 라이브러리에서 제공하는 주석으로, JSON 직렬화 시에 해당 필드의 값을 어떻게 처리할지 지정할 수 있습니다. `JsonInclude.Include.NON_NULL`을 적용하면, 해당 필드의 값이 `null`이 아닌 경우에만 JSON으로 직렬화되며, `null`인 경우에는 직렬화하지 않습니다.

```java
import com.fasterxml.jackson.annotation.JsonInclude;

public class Book {
    private String title;
    private String author;
    
    // Getters, Setters, Constructors, toString 등은 생략
    
    @JsonInclude(JsonInclude.Include.NON_NULL)
    public String getAuthor() {
        return author;
    }
}
```

3. `@ToString`:
`@ToString`은 Lombok 프로젝트에서 제공하는 주석으로, 클래스에 적용할 경우 해당 클래스의 `toString` 메서드를 자동으로 생성해 줍니다. `toString` 메서드를 사용하여 객체의 내용을 문자열로 표현할 때 유용합니다.

```java
import lombok.ToString;

@ToString
public class Point {
    private int x;
    private int y;
}

// @ToString을 사용한 클래스는 자동으로 toString 메서드를 생성합니다.
```

`@XmlAccessorType`는 JAXB(Java Architecture for XML Binding) 라이브러리에서 XML 바인딩 작업을 수행할 때 사용되는 주석(annotation)입니다. 이 주석은 클래스의 필드 또는 프로퍼티가 XML 바인딩에 어떻게 사용되는지를 지정하는 데 사용됩니다.

JAXB는 Java 클래스와 XML 문서 사이의 매핑을 단순화하고 자동화하는 기능을 제공합니다. 클래스의 필드 또는 프로퍼티를 XML 요소 또는 XML 속성과 매핑하여 객체를 XML로 직렬화하거나 XML을 다시 객체로 역직렬화할 수 있습니다.

`@XmlAccessorType` 주석은 다음과 같은 값들 중 하나로 구성될 수 있습니다:

- `XmlAccessType.FIELD`: 이 값은 클래스의 필드를 XML 요소로 매핑합니다. 필드를 직접 노출하고, `Getter`와 `Setter`를 사용하지 않습니다.
- `XmlAccessType.PROPERTY`: 이 값은 클래스의 프로퍼티를 XML 요소로 매핑합니다. `Getter`와 `Setter` 메서드를 사용하여 프로퍼티를 접근합니다.
- `XmlAccessType.PUBLIC_MEMBER`: 이 값은 클래스의 모든 public 필드와 프로퍼티를 XML로 매핑합니다.
- `XmlAccessType.NONE`: 이 값은 어떤 필드나 프로퍼티도 XML로 매핑하지 않습니다.

기본적으로, `@XmlAccessorType`는 `XmlAccessType.PUBLIC_MEMBER`를 사용하며, 클래스의 모든 public 필드와 프로퍼티를 XML로 매핑합니다. 다음은 간단한 예시입니다:

```java
import javax.xml.bind.annotation.XmlAccessType;
import javax.xml.bind.annotation.XmlAccessorType;
import javax.xml.bind.annotation.XmlRootElement;

@XmlRootElement
@XmlAccessorType(XmlAccessType.FIELD)
public class Person {
    private String name;
    private int age;

    // Getters, Setters, Constructors 등은 생략
}
```

위의 예시에서 `@XmlAccessorType(XmlAccessType.FIELD)`는 `Person` 클래스의 필드를 XML 요소로 매핑한다는 의미입니다. `Getter`와 `Setter`를 사용하지 않고 필드를 직접 노출하여 XML 바인딩을 수행할 수 있습니다.

이렇게 `@XmlAccessorType` 주석을 사용하면 JAXB가 클래스와 XML 사이의 매핑 방법을 지정할 수 있으며, XML 바인딩 작업을 보다 편리하게 수행할 수 있습니다.

```java
public class Main {
    public static void main(String[] args) {
        int a = 5; // 이진수로 101
        int b = 3; // 이진수로 011

        System.out.println(result); // 출력: 6 (이진수로 110)

}
```

---

### 🔹 2. **매개변수 (Parameter)**
- 메서드를 정의할 때 **메서드가 받을 값**을 임시로 담는 **변수**입니다.
- 즉, 메서드의 **입력값을 받아들이는 변수 이름**입니다.
- 예제에서 `int methodA(int a, int b)`에서 **`a`와 `b`**가 **매개변수**입니다.
- 메서드가 호출될 때, 호출하는 쪽에서 전달한 값이 이 매개변수에 복사됩니다.

---

### 🔹 3. **인수 (Argument)**  
- 메서드를 **호출할 때 실제로 전달하는 값**입니다.
- 예제에서 `methodA(x, y)`를 호출할 때 **`x`와 `y`**가 **인수**입니다.  
- 즉, **매개변수에 복사될 실제 데이터**입니다.

---

### ⚡ 비유로 정리
| 구분 | 코드 예시 | 역할 | 비유 |
|------|---------|------|------|
| **변수** | `int x = 5;` | 데이터를 저장하는 공간 | 컵 자체 |
| **인수** | `methodA(x, y)` | 메서드를 호출할 때 전달하는 값 | 컵 안에 실제로 담은 물 |
| **매개변수** | `int methodA(int a, int b)` | 인수를 받아 저장하는 메서드 내부 변수 | 컵을 받는 사람의 손 |

---

✅ **중요한 차이**  
- **인수**는 **호출 시점**에서 존재하는 값(실제 데이터).  
- **매개변수**는 메서드가 **정의될 때** 선언되는 값(데이터를 받을 그릇).  
- 결국 호출 시 **인수 → 매개변수**로 값이 복사됩니다.

### 💡 기본 개념

- **구현 클래스** = 그 계약을 실제 코드로 작성한 클래스

---

### ✅ 인터페이스를 구현하면서 변수를 가지는 클래스
예시를 봅시다.

```java
interface Animal {
    void sound(); // 반드시 구현해야 하는 메서드
}

class Dog implements Animal {
    private String name; // ✅ 인스턴스 변수

    public Dog(String name) {
        this.name = name;
    }

    @Override
    public void sound() {
        System.out.println(name + "가 멍멍!");
    }
}
```

여기서 `Dog` 클래스는:
- **인터페이스 `Animal`을 구현**했기 때문에 `sound()` 메서드를 반드시 작성해야 함.
- 동시에 **자기 자신의 상태를 저장할 변수(`name`)를 가짐**.

---

### 🔑 핵심 포인트
| 구분 | 의미 |
|------|------|
| **일반 클래스** | 메서드/변수를 자유롭게 정의할 수 있는 클래스 |
| **인터페이스 구현 클래스** | **일반 클래스이면서** 특정 인터페이스의 계약을 반드시 지켜야 하는 클래스 |
| **추상 클래스** | 직접 객체 생성은 불가, 상속을 통해 완성해야 함 |

➡️ 즉, **인터페이스를 구현했다고 해서 “평범한 클래스가 아니다”라고 할 수는 없음**.  
다만 **"역할(계약)"을 가진 클래스**가 된 것이라고 보는 게 정확합니다.

---

### ⚡ 비교 예시
| 클래스 종류 | 변수 보유 가능? | 객체 생성 가능? | 특징 |
|-------------|----------------|----------------|------|
| **일반 클래스** | ✅ 가능 | ✅ 가능 | 제한 없음 |
| **인터페이스** | ⚠️ `static final` 상수만 가능 | ❌ 불가능 | 구현을 강제 |
| **인터페이스 구현 클래스** | ✅ 가능 | ✅ 가능 | 일반 클래스지만 특정 기능을 반드시 구현해야 함 |
| **추상 클래스** | ✅ 가능 | ❌ (상속 필요) | 일부 구현 + 강제 규칙 |

### ✅ 1. **변수(Variable)**  
> 데이터를 담는 **그릇 자체**.

- **예시:** `int x = 5;`, `int y = 10;` → `x`, `y`는 **변수**.
- 메모리에 실제 값(5, 10)을 저장할 **공간**이 생깁니다.

---

### ✅ 2. **인수(Argument)**  
> 메서드를 호출할 때 **실제로 전달되는 값**.

- **예시:** `methodA(x, y);` → 여기서 `x`, `y`는 **인수**.
- 호출 시점에 **x와 y 안에 들어 있는 값(5, 10)** 이 **매개변수**로 복사됩니다.
- 즉, **값(데이터) 그 자체**가 넘어갑니다.

---

### ✅ 3. **매개변수(Parameter)**  
> 메서드가 **값을 받을 준비를 하는 변수 이름**.

- **예시:** `int methodA(int a, int b)` → `a`, `b`는 **매개변수**.
- 메서드를 **정의할 때** 미리 자리만 만들어두는 변수입니다.
- 호출될 때 **인수 값이 복사되어 들어옵니다**.

---

### ⚡ 흐름으로 이해
| 단계 | 코드 | 역할 | 상태 |
|------|------|------|------|
| **변수 선언** | `int x = 5;` | 값을 저장 | `x` 공간에 `5` 저장 |
| **메서드 호출(인수 전달)** | `methodA(x, y);` | 값(5, 10) 전달 | **인수**는 `x`의 값 `5`, `y`의 값 `10` |
| **메서드 정의(매개변수)** | `int methodA(int a, int b)` | 값 받을 준비 | `a`에 5 복사, `b`에 10 복사 |

---

### 🔑 핵심 정리
- `변수` → 값을 **저장하는 메모리 공간** (`x`, `y`).
- `인수` → 메서드 호출 시 **넘어가는 실제 값** (x에 저장된 5, y에 저장된 10).
- `매개변수` → 메서드가 인수를 **받기 위해 준비된 이름** (`a`, `b`).

---

👉 **당신이 쓴 요약**  
> int x = 5; → **변수**  
> methodA(x, y); → `x`, `y`는 **인수(실제 값이 들어 있는 상태)**  
> int methodA(int a, int b) → `a`, `b`는 **매개변수(값을 받을 준비만 한 상태)**  

✅ **완벽하게 맞는 설명**입니다. 👏

### 💡 상황 정리
지금 구조를 예제로 코드로 표현하면 대략 이렇게 된 거죠?

```java
interface Remote {
    void powerOn();
}

class RemoteA implements Remote {
    @Override
    public void powerOn() {
        System.out.println("RemoteA 켜짐");
    }
}

class RemoteB implements Remote {
    @Override
    public void powerOn() {
        System.out.println("RemoteB 켜짐");
    }
}

class RemoteC implements Remote {
    @Override
    public void powerOn() {
        System.out.println("RemoteC 켜짐");
    }
}

// Remote 타입을 멤버 변수로 가지는 클래스
class Controller {
    private Remote remote; // ✅ 인터페이스 타입

    public Controller(Remote remote) { // ✅ 생성자에서 인터페이스 타입으로 받음
        this.remote = remote;
    }

    public void turnOn() {
        remote.powerOn();
    }
}
```

이제 이렇게 사용할 수 있습니다:

```java
public class Main {
    public static void main(String[] args) {
        Controller c1 = new Controller(new RemoteA()); // ✅ RemoteA 가능
        Controller c2 = new Controller(new RemoteB()); // ✅ RemoteB 가능
        Controller c3 = new Controller(new RemoteC()); // ✅ RemoteC 가능

        c1.turnOn(); // RemoteA 켜짐
        c2.turnOn(); // RemoteB 켜짐
        c3.turnOn(); // RemoteC 켜짐
    }
}
```

### ✅ 왜 가능한가? → **다형성(Polymorphism)**
- **인터페이스 타입의 변수**는 **그 인터페이스를 구현한 어떤 클래스의 객체도 참조할 수 있음**.
- `Controller`의 생성자는 **Remote** 타입을 받지만,
  `RemoteA`, `RemoteB`, `RemoteC`는 모두 `Remote`를 구현했기 때문에  
  `new RemoteA()`, `new RemoteB()`, `new RemoteC()` 전부 인자로 전달할 수 있음.

➡️ **즉, Remote remote = new RemoteA();** 이런 식으로 인터페이스 타입에 구현체를 담을 수 있음.

| 타입 | 실제 객체 | 가능 여부 |
|------|-----------|----------|
| `Remote` | `new RemoteA()` | ✅ |
| `Remote` | `new RemoteB()` | ✅ |
| `Remote` | `new RemoteC()` | ✅ |

### ⚡ 한 줄 정리
> **Remote** 인터페이스를 매개변수로 받는 생성자라면,  
> `RemoteA`, `RemoteB`, `RemoteC` 중 어느 객체를 넣어도 된다.  
> 이는 **인터페이스 다형성** 덕분에 가능한 것이다.

## ✅ 1. **리모컨(Controller) → 실제 기기(Device)**  (가장 직관적인 예)
당신의 `Remote` 예제가 사실 **실무에도 딱 맞는 패턴**이에요.

### 시나리오
- TV, 에어컨, 오디오 등 **다양한 기기**를 제어해야 하는 `UniversalController`를 만든다고 가정.
- 각 기기는 켜는 방식이 다름.
- 하지만 컨트롤러 입장에서는 **"켜라"** 라는 명령만 있으면 됨.

### 코드
```java
interface Remote {
    void powerOn();
}

class TVRemote implements Remote {
    public void powerOn() { System.out.println("TV 켜짐"); }
}

class AirconRemote implements Remote {
    public void powerOn() { System.out.println("에어컨 켜짐"); }
}

class UniversalController {
    private Remote remote;

    public UniversalController(Remote remote) {
        this.remote = remote;
    }

    public void clickPower() {
        remote.powerOn();
    }
}

public class Main {
    public static void main(String[] args) {
        UniversalController c1 = new UniversalController(new TVRemote());
        UniversalController c2 = new UniversalController(new AirconRemote());

        c1.clickPower(); // TV 켜짐
        c2.clickPower(); // 에어컨 켜짐
    }
}
```

➡️ **장점**:  
- 새로운 기기(냉장고, 전등 등)가 추가되더라도  
  `Remote` 인터페이스만 구현하면 **컨트롤러 코드 수정 없이** 확장 가능.

## ✅ 2. **결제 시스템 (Payment Gateway)** – 💳 실무에서 정말 자주 나옴
여러 결제 수단(카드, 카카오페이, 네이버페이 등)을 지원해야 할 때.

### 코드
```java
interface Payment {
    void pay(int amount);
}

class CreditCard implements Payment {
    public void pay(int amount) {
        System.out.println(amount + "원 신용카드 결제 완료");
    }
}

class KakaoPay implements Payment {
    public void pay(int amount) {
        System.out.println(amount + "원 카카오페이 결제 완료");
    }
}

class PaymentProcessor {
    private Payment payment;

    public PaymentProcessor(Payment payment) {
        this.payment = payment;
    }

    public void process(int amount) {
        payment.pay(amount);
    }
}
```

### 사용
```java
PaymentProcessor p1 = new PaymentProcessor(new CreditCard());
p1.process(10000); // 10000원 신용카드 결제 완료

PaymentProcessor p2 = new PaymentProcessor(new KakaoPay());
p2.process(15000); // 15000원 카카오페이 결제 완료
```

➡️ **장점**:
- 새로운 결제 방식이 추가되어도 `PaymentProcessor` 수정 필요 없음.
- `Payment` 인터페이스만 구현하면 됨.

```java

    public void log(String message) {
        System.out.println("파일에 저장: " + message);
    }
}

class DatabaseLogger implements Logger {
    public void log(String message) {
        System.out.println("DB에 저장: " + message);
    }
}

class LogService {
    private Logger logger;

    public LogService(Logger logger) {
        this.logger = logger;
    }

    public void writeLog(String msg) {
        logger.log(msg);
    }
}
```

➡️ **장점**:  
- 운영 환경이 바뀌어 로그를 **DB → 클라우드**로 전환해야 해도,  
  `LogService`는 코드 변경 없이 새 Logger만 전달하면 됨.

---

## 🔑 핵심 패턴: **전략 패턴(Strategy Pattern)**

위 모든 예제는 사실상 **전략 패턴(Strategy Pattern)**의 일종이에요.

- **공통된 동작**(리모컨 동작, 결제, 로그 저장)을 인터페이스로 정의.
- **구체적 전략**(TV, 카드결제, 파일저장)은 클래스에서 구현.
- 실행 시점에서 어떤 전략을 쓸지 **주입(Injection)**.

➡️ 이 덕분에 코드가 **확장에 열려(Open)** 있고, **수정에는 닫혀(Closed)** 있게 설계할 수 있음.  
(→ **OCP: 개방-폐쇄 원칙**, SOLID의 핵심)

## 🔹 실무 버전으로 상상해보자
아래는 실무에서 `FileLogger`와 `DatabaseLogger`가 실제로 하게 될 일의 차이를 조금 더 현실적으로 풀어본 예시입니다.

---

### 1️⃣ **FileLogger (파일에 로그 저장)**  
파일에 로그를 남기려면:
- 파일 경로 지정
- `FileWriter` 또는 `BufferedWriter` 사용
- 파일을 열고(`open`), 쓰고(`write`), 닫아야(`close`) 함

```java
class FileLogger implements Logger {
    private BufferedWriter writer;

    public FileLogger(String filePath) {
        try {
            writer = new BufferedWriter(new FileWriter(filePath, true));
        } catch (IOException e) {
            e.printStackTrace();
        }
    }

    @Override
    public void log(String message) {
        try {
            writer.write(message + "\n");
            writer.flush(); // 파일에 즉시 반영
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

➡️ 이 클래스는 **파일 IO 라이브러리(java.io)** 를 직접 사용하며,
실제 파일에 데이터를 저장합니다.

---

### 2️⃣ **DatabaseLogger (DB에 로그 저장)**  
DB에 로그를 남기려면:
- JDBC 또는 JPA를 사용해 DB 연결
- SQL INSERT 실행
- 연결을 닫고 예외 처리

```java
class DatabaseLogger implements Logger {
    private Connection conn;

    public DatabaseLogger() {
        try {
            conn = DriverManager.getConnection(
                "jdbc:mysql://localhost:3306/mydb", "user", "password"
            );
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }

    @Override
    public void log(String message) {
        String sql = "INSERT INTO logs (message) VALUES (?)";
        try (PreparedStatement stmt = conn.prepareStatement(sql)) {
            stmt.setString(1, message);
            stmt.executeUpdate();
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }
}
```

➡️ 이 클래스는 **JDBC 라이브러리(java.sql)** 를 사용해서,
데이터베이스에 데이터를 저장합니다.

---

### ✅ 중요한 차이점
| 구분 | FileLogger | DatabaseLogger |
|------|------------|-----------------|
| 저장 위치 | 로컬 파일 시스템 | 데이터베이스 |
| 필요한 라이브러리 | `java.io.*` | `java.sql.*` (JDBC 등) |
| 동작 방식 | 파일 열기 → 내용 쓰기 → 닫기 | DB 연결 → SQL 실행 → 닫기 |
| 에러 상황 | 파일 권한 문제, 디스크 공간 부족 | DB 연결 실패, SQL 오류 |

➡️ 단순히 `System.out.println()`으로만 보여주면 이런 복잡함이 가려져서  
"둘이 뭐가 다르지?"라는 생각이 드는 게 당연해요.

---

## 🔑 그래서 인터페이스가 중요한 이유
이제 왜 `Logger` 인터페이스가 필요한지가 **확** 드러납니다.

`LogService`는 이렇게 동작합니다:
```java
class LogService {
    private Logger logger;

    public LogService(Logger logger) {
        this.logger = logger;
    }

    public void writeLog(String msg) {
        logger.log(msg);
    }
}
```

- `LogService`는 **파일을 어떻게 쓰는지**, **DB에 어떻게 연결하는지** 전혀 몰라도 됨.
- 오직 `logger.log(msg)`만 부르면 됨.

➡️ **결과적으로**
- 파일 로깅 → `new FileLogger("log.txt")` 로 교체
- DB 로깅 → `new DatabaseLogger()` 로 교체  
이렇게만 하면 `LogService`는 코드 한 줄도 수정하지 않고 **저장 방식을 갈아낄 수 있음**.

---

## ⚡ 실무에서 진짜 이렇게 쓰는 곳
이 패턴은 **Java Logger 프레임워크**에서 그대로 쓰입니다.
- 예: `SLF4J` + `Logback` + `Log4j`
  ```java
  Logger logger = LoggerFactory.getLogger(MyClass.class);
  logger.info("Hello World");
  ```
- 개발자는 `logger.info()`만 부르면 됨.
- 실제 저장은 **ConsoleLogger**, **FileLogger**, **DatabaseLogger**, **ElasticSearchLogger** 등 설정 파일만 바꿔서 결정.

## 💡 핵심 정리
당신이 느낀 **"이렇게 나눌 필요가 있나?"** 라는 의문은,
제가 보여준 `println` 수준의 간단한 예제 때문에 생긴 겁니다.  
실제 코드에서는:

✅ **FileLogger** → 파일을 여는 라이브러리, 예외처리, 디스크 I/O  
✅ **DatabaseLogger** → JDBC 연결, SQL 실행, 트랜잭션 처리  

이렇게 구현체마다 **필요한 기술이 완전히 다르기 때문에**,  
인터페이스로 묶어두면 **메인 로직이 특정 기술에 종속되지 않게 만들 수 있는 것**입니다.
