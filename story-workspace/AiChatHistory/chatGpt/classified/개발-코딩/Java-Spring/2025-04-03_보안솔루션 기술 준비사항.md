### 🔹 **해시 함수(Hash Function)의 역할**  
해시 함수는 **입력값(비밀번호 등)을 고정된 길이의 값(해시 값)으로 변환하는 함수**야. 중요한 특징은:  
1. **단방향성** → 해시 값에서 원래 값을 복원할 수 없음.  
2. **같은 입력 → 같은 해시 값**  
3. **다른 입력 → 다른 해시 값(충돌 가능성이 극도로 낮음)**  
4. **작은 변화에도 완전히 다른 결과 출력 (Avalanche 효과)**  

---

### 🔹 **로그인 과정에서의 해시 함수 활용**
로그인할 때 비밀번호를 해시 함수로 변환하여 저장된 값과 비교하는 방식이야.  

1️⃣ **회원가입 시**  
   - 사용자가 `"password123"` 같은 비밀번호를 입력하면  
   - 서버는 **해시 함수(SHA-256 등)를 적용해서 변환** → `ef92b778baf…` 같은 값으로 저장  
   - (보안을 강화하기 위해 **Salt**라는 랜덤 값을 추가해서 해싱하기도 함)  

2️⃣ **로그인 시 검증 과정**  
   - 사용자가 `"password123"`을 입력하면  
   - 서버는 동일한 해시 함수를 적용해서 변환  
   - 저장된 해시 값(`ef92b778baf…`)과 비교  
   - 값이 같으면 **로그인 성공**, 다르면 실패  

---

### 🔹 **해시와 암호화의 차이**
- **해시(Hashing)**:  
  - 단방향 변환 (복호화 불가능)  
  - 주로 비밀번호 저장에 사용  
  - SHA-256, SHA-512, bcrypt, PBKDF2 등  

- **암호화(Encryption)**:  
  - 양방향 변환 (복호화 가능)  
  - 데이터 보호(전송 시 보안 유지) 목적  
  - AES, RSA 등 사용  

---

### 🔹 **보안 강화를 위한 추가 기법**
1. **Salt & Pepper 기법**  
   - 해시 값이 패턴화되는 걸 방지하기 위해 **랜덤한 값(Salt)을 추가**  
   - `"password123"` → `"randomSalt+password123"` → 해시 적용  

2. **Key Stretching (bcrypt, PBKDF2, Argon2)**  
   - 해시를 여러 번 반복 계산하여 **연산을 무겁게 만들어** 공격을 어렵게 함  

---

### 🔹 **디코딩? (De... 어쩌구였던 부분?)**
- 해시는 원래 **복호화(Decode, Decrypt) 불가능**  
- 하지만, 공격자가 **무작위 조합을 대입(Brute-force, Rainbow Table Attack)** 해서 같은 해시 값을 찾으려 할 수 있음  
- 이를 막기 위해 Salt, bcrypt 같은 기법을 사용  

---

### 🔹 **결론**
✅ 해시 함수는 **로그인 과정에서 비밀번호를 안전하게 저장하고 검증**하는 역할을 함  
✅ 암호화(Encryption)와는 다르게, 해시는 **복호화(Decrypt)할 수 없음**  
✅ Salt, bcrypt 같은 보안 기법을 활용하면 공격을 더 어렵게 만들 수 있음

### 🔹 **복호화(Decrypt)가 불가능한 이유**
- 일반적인 **암호화(Encryption)** 방식은 **비밀키(Private Key) 또는 공개키(Public Key)**를 사용해서 복호화할 수 있음.  
- 하지만 **해시 함수(Hash Function)**는 **단방향(one-way) 함수**라서,  
  - 입력값이 같으면 항상 같은 해시 값을 출력하지만  
  - 해시 값에서 원래 입력값을 직접 복원하는 건 불가능!  

💡 **즉, 복호화할 수 없는 이유는 공격자가 해시 생성 방식을 알아도 역추적할 방법이 없기 때문**  

---

### 🔹 **Brute-force(무차별 대입 공격) 대응 방법**
공격자가 특정 아이디의 **비밀번호 길이나 일부 문자**를 알고 있다면, 무작위로 가능한 모든 조합을 시도해서 로그인할 수도 있어. 이를 막기 위해 여러 보안 조치가 필요해.  

✅ **1. 로그인 제한 & 지연 처리**  
   - **일정 횟수 실패 시 계정 잠금 또는 일시 정지**  
     - 예: 5회 이상 틀리면 10분 동안 로그인 차단  
   - **점진적 지연 처리** (Exponential Backoff)  
     - 실패 횟수에 따라 로그인 응답 시간을 길게 조정  
     - 예: 1회 실패 → 1초 대기, 2회 실패 → 2초 대기, 3회 실패 → 4초 대기  

✅ **2. CAPTCHA 적용**  
   - 자동화 공격을 막기 위해 로그인 시도 횟수가 많아지면 CAPTCHA(이미지 인증, reCAPTCHA) 요구  

✅ **3. 2단계 인증 (MFA, Multi-Factor Authentication)**  
   - 로그인할 때 **추가 인증(OTP, 이메일, SMS 인증 코드 등)**을 요구  
   - 비밀번호가 유출되더라도 쉽게 로그인하지 못하도록 차단  

✅ **4. 강력한 비밀번호 정책 적용**  
   - 특정 패턴(예: `123456`, `password`)은 사용 금지  
   - 대문자, 소문자, 숫자, 특수문자를 포함하도록 요구  
   - 일정 주기마다 비밀번호 변경 요구 (단, 너무 자주 강제하면 보안성 오히려 낮아질 수 있음)  

✅ **5. 해시 + Salt + Key Stretching**  
   - **Salt**: 같은 비밀번호라도 다른 해시 값이 나오도록 랜덤한 문자열 추가  
   - **Key Stretching (bcrypt, PBKDF2, Argon2)**: 해시 연산을 여러 번 반복하여 연산량을 늘려 공격을 어렵게 만듦  

---

### 🔹 **결론**
1️⃣ **해시는 단방향이므로 복호화할 수 없고, 대신 무작위 대입(Brute-force) 공격이 문제**  
2️⃣ **Brute-force 공격을 막으려면 로그인 제한, CAPTCHA, 2단계 인증 등이 필요**

## 🔹 **기본적인 CAPTCHA 방식**
### 1️⃣ **전통적인 CAPTCHA** (그림 맞추기, 문자인식)
- 사용자가 **왜곡된 문자**를 읽거나 **특정 이미지를 고르는 방식**  
- ✅ **장점**: 자동화된 봇(스크립트)에 대한 방어 가능  
- ❌ **단점**: 사람이 직접 입력해야 해서 **사용자 경험(UX)이 나빠짐**  

### 2️⃣ **reCAPTCHA (구글)**  
- Google에서 만든 CAPTCHA 시스템  
- **버전 2 (reCAPTCHA v2)** → "신호등, 횡단보도를 선택하세요" 같은 이미지 선택  
- **버전 3 (reCAPTCHA v3)** → **사용자 행동 분석**으로 자동 판별 (클릭 필요 없음)

## 🔹 **reCAPTCHA v3: 어떻게 사람인지 구별할까?**
최근에는 **"이 사람은 정상적인 사용자일까?"**를 자동으로 판별하는 방식(reCAPTCHA v3)이 많아졌어.  

### ✅ **1. 사용자 행동 기반 분석**
- 마우스 움직임, 스크롤 패턴, 클릭 속도 등을 분석  
- 사람과 봇의 행동 패턴 차이를 학습하여 판별  

### ✅ **2. 사용자의 인터넷 활동(쿠키, 브라우징 기록) 분석**
- Google 계정 로그인 여부, 브라우징 이력 등을 참고  
- 사람이 실제로 여러 웹사이트를 방문하고 활동했다면 신뢰도 상승  
- 브라우징 이력이 없거나, 프록시/VPN 등을 쓰면 의심됨  

### ✅ **3. "점수 기반" 자동 판별 (No CAPTCHA)**
- reCAPTCHA v3는 0~1 사이의 점수를 부여  
  - 1.0: 확실한 인간  
  - 0.0: 확실한 봇  
  - 중간 점수일 경우 추가 보안 절차 요구  

이 방식 덕분에 요즘은 **별도의 CAPTCHA 입력 없이도 로그인이나 폼 제출이 가능**한 경우가 많아!

## 🔹 **사용자 친화적인 CAPTCHA 대체 방법**
CAPTCHA는 사용자 경험(UX)이 불편하기 때문에, 다음과 같은 방식으로 대체되기도 해.  

### ✅ **1. 이메일/SMS 인증 (2FA, OTP)**
- 특정 행동(로그인, 결제 등) 시 **추가 인증 코드 입력** 요구  
- 사람이 직접 받을 수 있는 이메일/SMS를 통해 인증하므로 보안성 높음  

### ✅ **2. WebAuthn (생체 인증)**
- 지문, 얼굴 인식, 보안 키(U2F) 등을 활용해 로그인  
- 최근 FIDO2(WebAuthn) 표준이 보급되면서 점점 더 많이 사용됨

## 🔹 **Invisible CAPTCHA (보이지 않는 CAPTCHA)**
기존의 **CAPTCHA(그림 맞추기, 문자 입력)**는 사용자 경험(UX)에 악영향을 줬는데, 이를 개선한 방식이 **Invisible CAPTCHA (보이지 않는 CAPTCHA)**야.  

### ✅ **Invisible CAPTCHA의 핵심 원리**
1. 사용자가 **아무것도 입력하지 않아도 됨**  
2. 사용자가 특정 동작(버튼 클릭, 페이지 로딩 등)을 할 때 **자동으로 보안 검증 진행**  
3. **점수가 높으면 CAPTCHA 없이 바로 진행**, 점수가 낮으면 CAPTCHA를 띄움  

👉 즉, **사용자는 아무것도 못 느끼고, 봇만 차단되는 방식**  

---

## 🔹 **어떤 기술이 쓰이나?**
### 1️⃣ **reCAPTCHA v3 (구글)**
- 사용자가 클릭하지 않아도 **백그라운드에서 자동으로 분석**  
- **"이 사용자가 실제 사람인지?"**를 행동 데이터 기반으로 점수 매김  
- 보통 **로그인 버튼이나 제출 버튼을 누를 때 자동 체크**  
- 점수가 낮으면 **추가 CAPTCHA(그림 선택) 표시**  

👉 **사용자 경험이 향상되고, 봇은 차단됨**  

### 2️⃣ **Honeypot CAPTCHA (허니팟 기법)**
- 폼(form) 안에 **사용자가 볼 수 없는 숨겨진 입력 필드 추가**  
- 정상적인 사용자는 입력하지 않지만, **자동화된 봇은 실수로 입력**  
- **입력된 경우 자동 차단**  

👉 사용자는 CAPTCHA를 안 봐도 되지만, 봇은 쉽게 걸려듦  

---

## 🔹 **Invisible CAPTCHA를 우회하는 봇도 존재?**
그렇긴 해.  
- 일부 봇은 **사용자의 마우스 클릭을 자동화해서 reCAPTCHA v3를 속이려 함**  
- 그래서 보안 강화 방법으로 **"행동 패턴 분석" + "브라우징 이력 체크"** 같은 추가 기술이 도입됨  

예를 들어:
1. **마우스 움직임, 클릭 타이밍 분석**  
   - 사람이 클릭하는 속도와 **봇이 클릭하는 속도는 차이가 있음**  
   - 봇은 너무 일정한 패턴이라 쉽게 탐지됨  

2. **브라우징 히스토리 & 쿠키 체크**  
   - 진짜 사용자는 **이전에 여러 사이트를 방문한 기록이 있음**  
   - 하지만 봇은 깨끗한 환경에서 동작하므로 탐지됨  

3. **디지털 지문(Fingerprinting) 분석**  
   - IP, 브라우저 설정, 화면 해상도 같은 데이터를 분석해서 동일한 봇이 반복해서 시도하는지 확인

2️⃣ **reCAPTCHA v3는 사용자 행동 + 브라우징 데이터를 분석해 점수를 매김**

# 🔹 **1. 초기 암호화: 대칭키 암호화 (Symmetric Encryption)**
과거에는 **대칭키 암호화** 방식이 일반적이었어.  
이 방식에서는 **암호화할 때 쓴 키로 복호화도 가능**해.  

### 🔑 **대칭키 암호화 과정**
1. 사용자가 입력한 비밀번호를 **하나의 키(Key)로 암호화**  
2. 저장된 데이터는 이 키를 사용해 **복호화 가능**  

### ❌ **문제점**
- **공격자가 키를 알게 되면 복호화가 가능** → 데이터 탈취 위험  
- **키를 안전하게 저장하는 것이 어려움**  

### ▶ **대표적인 대칭키 암호화 방식**
- **DES (Data Encryption Standard)** → 56비트 키 (현재는 취약)  
- **AES (Advanced Encryption Standard)** → 128/192/256비트 키 (현대 표준)  

이 방식은 지금도 파일 암호화 같은 곳에서는 쓰이지만,  
**비밀번호 저장**에는 더 강력한 방법이 필요했어.

---

# 🔹 **2. 비대칭키 암호화 (Asymmetric Encryption)**
**공개키(Public Key)와 비밀키(Private Key)**를 사용하는 방식이야.  
- **공개키(Public Key)**: 암호화를 위한 키 (모두에게 공개)  
- **비밀키(Private Key)**: 복호화를 위한 키 (본인만 보유)  

### 🔑 **비대칭키 암호화 과정**
1. 서버가 **공개키(Public Key)**를 사용해 데이터를 암호화  
2. 복호화는 **비밀키(Private Key)**를 가진 사람만 가능  

### ✅ **장점**
- **비밀번호를 직접 저장하지 않아도 됨**  
- **서버는 공개키만 보관 → 비밀키가 유출되지 않으면 안전**  

### ❌ **단점**
- **연산량이 많아서 속도가 느림**  

### ▶ **대표적인 비대칭 암호화 방식**
- **RSA (현재까지 널리 사용됨, 2048비트 이상 권장)**  
- **ECC (Elliptic Curve Cryptography, RSA보다 효율적)**  

**비대칭키 방식은 로그인 과정에서 데이터를 안전하게 주고받는 데 사용됨**  
하지만 **비밀번호 저장**에는 더 좋은 방식이 필요했어.  

---

# 🔹 **3. 단방향 해시 함수 (One-way Hash Function)**
비밀번호 저장을 더 안전하게 하기 위해,  
**비밀번호를 복호화할 수 없는 형태(단방향)로 변환하는 방식**이 등장했어.  

### 🔑 **해시 함수의 특징**
- 입력값(비밀번호)을 **일정한 길이의 해시값으로 변환**  
- 해시값은 **복호화가 불가능함** (즉, 원래 비밀번호를 알아낼 수 없음)  

### ✅ **장점**
- **설사 데이터베이스가 유출되더라도, 원본 비밀번호를 알 수 없음**  
- **비밀번호를 비교할 때만 해시값을 이용해서 확인 가능**  

### ❌ **단점**
- **동일한 비밀번호는 항상 같은 해시값을 가짐 → 공격에 취약**  
  (예: "password123" → 해시값이 같으면 공격자가 유추 가능)  

### ▶ **대표적인 해시 알고리즘**
- **MD5** (이전에는 많이 쓰였지만 취약함)  
- **SHA-1** (취약점 발견됨 → 현재 비추천)  
- **SHA-256, SHA-512** (비교적 안전)  

하지만 단순한 해시만으로는 안전하지 않았어.  
그래서 **"소금(salt)"** 기법이 추가되었어.

---

# 🔹 **4. Salt + Hash (솔트 기법)**
동일한 비밀번호라도 **각각 다른 해시값을 가지게 만드는 방법**이야.  

### 🔑 **Salt 방식**
1. **비밀번호에 무작위 문자열(Salt)을 추가**  
2. **이 값을 해시 함수에 넣어서 해시값을 생성**  
3. **Salt 값도 함께 저장** (비밀번호 자체는 저장하지 않음)  

### ✅ **장점**
- **같은 비밀번호라도 해시값이 다르게 저장됨 → 공격에 강함**  
- **사전 공격(Dictionary Attack) 방어 가능**  

### ▶ **예시**
```plaintext
비밀번호: "password123"
Salt: "Xj12KlmN"
Hash(Salt + 비밀번호) = "a7f6c9..."
```
이렇게 저장하면, **공격자가 해시값을 보고도 원본을 유추할 수 없음**  

---

# 🔹 **5. Key Derivation Function (KDF, 키 유도 함수)**
더 강력한 보안을 위해 **해시 연산을 반복**해서 계산하는 방식이 나왔어.  
이렇게 하면 **브루트포스 공격(무차별 대입 공격)을 매우 어렵게 만들 수 있어.**  

### 🔑 **KDF 방식**
1. **비밀번호 + Salt 값을 여러 번 해시 처리**  
2. 연산량이 많아서 **공격자가 쉽게 크랙(해독)할 수 없음**  

### ✅ **대표적인 KDF 알고리즘**
- **PBKDF2** (가장 많이 사용됨)  
- **bcrypt** (솔트가 자동 포함됨, 보안성이 높음)  
- **Argon2** (가장 최신 알고리즘, 최우수 보안 평가 받음)  

### ▶ **bcrypt 예제**
```plaintext
비밀번호: "password123"
bcrypt 해시값: "$2a$12$A8qE5K..."
```
- bcrypt는 **자동으로 솔트 추가**  
- 연산량을 조절해서 **컴퓨팅 성능이 좋아져도 보안 유지 가능**  

현재 **비밀번호 저장 방식으로 가장 추천되는 방법이 bcrypt나 Argon2**야.  

---

# 🔹 **결론: 암호화 방식의 발전 과정**
1️⃣ **초기에는 대칭키 암호화(DES, AES) → 복호화 가능하므로 취약**  
2️⃣ **비대칭키 암호화(RSA, ECC) → 로그인 과정에서는 유용하지만 저장에는 부적절**  
3️⃣ **단방향 해시 함수(SHA-256 등) → 복호화 불가능하지만, 동일한 값에 취약**  
4️⃣ **Salt + Hash → 같은 비밀번호라도 다른 해시값으로 저장하여 보안 강화**  
5️⃣ **Key Derivation Function (bcrypt, Argon2) → 해시 연산을 반복하여 공격 어렵게 만듦**  

💡 **현재 가장 추천되는 비밀번호 저장 방법은 bcrypt 또는 Argon2 사용**  
이제는 비밀번호를 **암호화해서 저장하는 것이 아니라, 해시 처리하여 복호화 자체를 불가능하게 만드는 방식**이 표준이 되었어! 🚀

## 🔹 **1. 해시 함수의 규칙성을 알아낼 수 있는 경우?**
해시 함수는 **입력값이 같으면 항상 같은 출력값을 생성**하기 때문에,  
어떤 방식으로든 규칙을 파악하면 **같은 해시값을 가지는 비밀번호를 찾을 수 있어.**  
하지만 이걸 **현실적으로 공격 가능한가?**가 중요해.  

다음과 같은 방법으로 해시값을 역추적하는 공격이 가능해.  

---

## 🔸 **2. 해시를 깨는 공격 방법**
💥 **(1) 사전 공격 (Dictionary Attack)**  
- 공격자가 **미리 많은 비밀번호를 해시 값으로 변환한 데이터베이스**를 만든 뒤,  
  데이터베이스에 있는 해시값과 비교해서 비밀번호를 알아내는 방식.  
- 예를 들어, 공격자가 `"password123"`의 해시값을 미리 알고 있다면,  
  같은 해시값이 나오면 비밀번호가 `"password123"`임을 알 수 있음.  

🛡 **방어 방법:**  
✅ **Salt 기법**을 사용해서 동일한 비밀번호라도 서로 다른 해시값을 갖게 만들면 효과적으로 방어할 수 있음.

---

💥 **(2) 레인보우 테이블 공격 (Rainbow Table Attack)**  
- 사전 공격을 더 효율적으로 만든 방식.  
- **미리 계산된 해시값 목록(레인보우 테이블)을 사용해서 빠르게 비밀번호를 찾아냄.**  
- 일반적인 사전 공격보다 훨씬 더 큰 데이터셋을 활용할 수 있음.  

🛡 **방어 방법:**  
✅ **Salt를 사용하면 같은 비밀번호라도 해시값이 달라지기 때문에 레인보우 테이블이 무용지물이 됨.**  
✅ 즉, Salt + Hash 기법을 사용하면 이 공격은 거의 무력화됨.  

---

💥 **(3) 브루트포스 공격 (Brute Force Attack)**  
- **가능한 모든 비밀번호를 하나씩 해싱해서 비교하는 방식**  
- 예를 들어, `"aaa"`, `"aab"`, `"aac"` 등 무작위로 입력값을 생성해서  
  해시값과 비교해 비밀번호를 알아내는 공격.  
- 해시 함수 자체의 규칙성을 분석하지 않아도, 단순 계산으로 무작정 시도하면 언젠가는 찾을 수 있음.  

🛡 **방어 방법:**  
✅ **KDF (bcrypt, Argon2) 같은 해시 함수 사용** → 계산 속도를 일부러 느리게 만들어 브루트포스 공격이 어려워짐.  
✅ **로그인 시도 제한** → 일정 횟수 이상 실패하면 계정 잠금.  

---

💥 **(4) 충돌 공격 (Collision Attack)**  
- 해시 함수는 **무한한 입력값을 유한한 길이로 변환하기 때문에, 이론적으로는 서로 다른 입력값이 같은 해시값을 가질 가능성이 있음.**  
- 이를 악용하면 **원래 비밀번호와 다른 입력값을 찾아서 같은 해시값을 만들 수 있음.**  
- 하지만 현대적인 보안 해시 함수(SHA-256, bcrypt 등)는 이 공격을 극도로 어렵게 만들어 놓음.  

🛡 **방어 방법:**  
✅ **SHA-256 이상, bcrypt, Argon2 같은 충돌 저항성이 높은 해시 함수 사용**  

---

## 🔹 **3. 해시 함수만으로는 완벽하지 않다!**
만약 **비밀번호를 단순 해시(SHA-256 같은 기본 해시)로만 저장한다면**,  
사전 공격이나 레인보우 테이블 공격에 취약할 수 있음.  
그래서 현재는 **단순 해시 사용을 피하고, 다음과 같은 방식으로 보완**하고 있어.  

✅ **Salt 추가 (무작위 문자열을 비밀번호에 추가해서 해싱)**  
✅ **bcrypt, Argon2 같은 KDF(키 유도 함수) 사용 → 연산을 느리게 만들어서 공격 어렵게 만듦**  
✅ **로그인 시도 제한, 2FA(2단계 인증) 도입 등 추가 보안 조치 적용**

✔ **Salt, KDF(예: bcrypt, Argon2) 같은 기술을 사용하면 현실적으로 공격이 거의 불가능해짐.**  
✔ 가장 중요한 건 **비밀번호를 안전하게 저장하는 것**이고,  
  **단순 해시만으로 보관하는 것은 보안상 위험하다.**  
✔ 그래서 **bcrypt, Argon2 같은 보안성이 높은 방법을 사용해야 한다!** 🚀

### 🔹 **Salt의 역할**  
**Salt는 각 사용자마다 다르게 생성되는 랜덤한 문자열**이야.  
즉, 같은 `"password"`를 입력해도 **각 계정마다 다른 해시값이 나오도록 만드는 역할**을 함.  

✔ 예를 들어, 두 개의 계정이 있다고 가정하자.  
✔ 같은 `"password"`를 비밀번호로 설정해도,  
✔ 각 계정에 **다른 Salt 값**을 추가하면 **서로 다른 해시값**이 생성됨.  

---

| User2 | password | `dsgsjfdsovncxkkn231` |

✔ 같은 비밀번호(`password`)는 **같은 해시값**을 가지게 됨.

✔ 따라서 **해커가 해시값을 보고 같은 비밀번호인지 유추하는 것이 불가능해짐!**

     `"password + Salt"` 조합을 직접 계산해야 함.  
   - Salt 없이 공격하면 `"password"` 하나만 해싱해서 비교하면 되지만,  
     **Salt가 있으면 수많은 조합을 직접 계산해야 해서 시간이 훨씬 오래 걸림.**  

2️⃣ **Salt는 해시값과 함께 저장되지만, 레인보우 테이블 공격이 불가능해짐**  
   - 레인보우 테이블 공격은 **미리 계산된 해시값 데이터베이스를 이용하는 공격**인데,  
     Salt를 적용하면 **각 계정마다 해시값이 달라져서 미리 계산된 값이 쓸모없어짐.**  

3️⃣ **Salt를 안전하게 관리하는 것도 중요함**  
   - 일반적으로 **Salt는 개별 계정마다 따로 저장되지만, 서버 내에서 안전하게 보호되어야 함.**  
   - 잘못 설계된 시스템에서는 모든 계정에 같은 Salt를 쓰기도 하는데,  
     그러면 보안이 약해짐. (좋은 설계는 **각 사용자별로 랜덤한 Salt를 부여하는 것!**)

## 🔹 **Salt가 유출된 경우, 해커는 어떻게 공격할까?**  

**📌 1. 브루트포스 공격 (Brute-force attack)**  
   - 해커가 **Salt 값과 해시된 비밀번호를 알고 있다면**,  
     `"입력한 비밀번호 + 해당 사용자의 Salt"` 조합을 계속 테스트하는 방식.  
   - 만약 `"password"`의 해시값이 `abc123`이라고 가정하면,  
     `"password" + Salt → 해시값`을 만들어 비교하는 식으로 시도 가능.  
   - **즉, Salt를 알고 있어도 비밀번호를 직접 알아내려면 여전히 해시값을 하나씩 대조해야 하므로 쉽지는 않음.**  
   - 다만, **비밀번호가 너무 짧거나 단순하면** 이 방법으로 쉽게 뚫릴 수 있음.

**📌 2. 사전 공격(Dictionary Attack)**  
   - 해커가 미리 수집한 **자주 사용되는 비밀번호 목록 (예: `password123`, `qwerty`, `12345678`)**을 기반으로,  
     Salt와 조합한 후 해시값을 비교하는 방식.  
   - 보통의 해시 함수(SHA-256 같은 것)를 사용하면, 이 방식도 **비밀번호가 단순하면 뚫릴 가능성이 있음.**  

**📌 3. GPU 병렬 연산을 이용한 대량 해시 계산**  
   - 현대의 고성능 GPU(예: RTX 3090)나 클러스터를 이용하면,  
     **수십억 개의 해시를 빠르게 계산 가능**해서 짧은 비밀번호라면 무력화될 수 있음.  
   - 특히, **해시 함수가 빠르게 계산되는 구조라면** 해커가 빠르게 조합을 시도할 수 있음.  
   - 따라서, SHA-256 같은 빠른 해시 함수 대신 **bcrypt, Argon2, PBKDF2 같은 연산량이 많은 해시 함수(KDF)**를 사용해야 안전함.

## 🔹 **그럼 Salt를 알아도 해킹이 어려운 이유는?**  

1️⃣ **비밀번호마다 해시값이 다르게 저장되므로, 레인보우 테이블 공격이 불가능함.**  
   - **Salt가 없으면**, 미리 계산된 해시값을 가진 **레인보우 테이블**을 이용해서  
     빠르게 비밀번호를 찾아낼 수 있음.  
   - 하지만 **Salt가 있으면**, 같은 비밀번호라도 해시값이 다 달라지기 때문에  
     **레인보우 테이블 공격을 무력화할 수 있음!**  

2️⃣ **해커가 Salt + 해시된 비밀번호를 알아도, 여전히 브루트포스를 해야 함.**  
   - 즉, **Salt가 있어도 "빠르게 비밀번호를 알아내는 방법"은 없음.**  
   - 무조건 `"비밀번호 + Salt"`를 하나하나 대조해야 해서,  
     비밀번호가 길고 복잡하면 뚫기 어려움.  

3️⃣ **KDF(키 유도 함수)를 사용하면 공격이 사실상 불가능해짐.**  
   - bcrypt, PBKDF2, Argon2 같은 KDF는 **일반 해시 함수보다 훨씬 연산량이 많고 느림.**  
   - 예를 들어 **SHA-256은 초당 수백만 개를 계산할 수 있지만, bcrypt는 초당 몇백 개밖에 계산 못함.**  
   - 즉, 해커가 **수십억 개의 조합을 대조하는 게 불가능에 가까워짐!**

## 🔹 **결론: Salt는 "공격 방법"을 바꾸지만, 해킹을 쉽게 만들지는 않는다!**  
✔ **Salt가 있다고 바로 비밀번호가 유출되는 건 아님.**  
✔ **Salt가 없으면 "같은 비밀번호"를 가진 계정을 쉽게 찾을 수 있지만, Salt가 있으면 불가능.**  
✔ **Salt가 있어도 해커는 브루트포스 공격을 시도할 수는 있지만, 비밀번호가 복잡하면 사실상 불가능.**  
✔ **더 강력한 보안을 위해서는 bcrypt, Argon2 같은 KDF를 사용해서 해싱해야 함.**

## 🔹 1. **PBKDF2 (Password-Based Key Derivation Function 2)**
- 반복 연산을 여러 번 수행해서 해커가 쉽게 브루트포스 공격을 못 하도록 만듦.
- N번 반복하면서 연산을 수행하면, 해커도 똑같이 N번 계산해야 함.
- **SHA-256 같은 빠른 해시 함수와 함께 사용함.**  

📌 **예제 (Python)**
```python
import hashlib
import os
from hashlib import pbkdf2_hmac

password = b"my_secure_password"  # 사용자의 비밀번호
salt = os.urandom(16)  # 랜덤한 salt 생성
iterations = 100_000  # 반복 횟수 (클수록 안전하지만 속도가 느려짐)

hashed_password = pbkdf2_hmac("sha256", password, salt, iterations)

print(f"Salt: {salt.hex()}")
print(f"Hashed Password: {hashed_password.hex()}")
```
**👉 특징:**  
✅ 반복 횟수(`iterations`)를 크게 하면 해킹 속도를 늦출 수 있음.  
✅ 보안성이 높지만, 연산량이 많아지면 로그인 속도가 느려질 수 있음.

## 🔹 2. **bcrypt**
- 내부적으로 `Blowfish` 암호 알고리즘을 기반으로 함.
- **자동으로 Salt를 포함해서 해싱**하므로 따로 관리할 필요 없음.
- `cost` 값을 조절하면 연산량을 조정할 수 있음.

📌 **예제 (Python)**
```python
import bcrypt

password = b"my_secure_password"

# 비밀번호를 해싱
hashed_password = bcrypt.hashpw(password, bcrypt.gensalt())

# 비밀번호 검증
if bcrypt.checkpw(password, hashed_password):
    print("로그인 성공!")
else:
    print("비밀번호 틀림!")

print(f"Hashed Password: {hashed_password.decode()}")
```
**👉 특징:**  
✅ 자동으로 `salt`를 추가함.  
✅ `cost` 값에 따라 연산량을 조절할 수 있음.  
✅ 해킹 시도가 매우 느려지므로 보안성이 높음.

## 🔹 3. **Argon2 (가장 강력한 KDF)**
- 2015년에 열린 **Password Hashing Competition (PHC)**에서 우승한 알고리즘.
- **메모리 사용량을 조절할 수 있어 GPU 병렬 연산 공격을 어렵게 만듦.**  
- 즉, **bcrypt보다도 강력한 보안성을 제공**함.

📌 **예제 (Python)**
```python
from argon2 import PasswordHasher

ph = PasswordHasher()

# 비밀번호 해싱
hashed_password = ph.hash("my_secure_password")

# 비밀번호 검증
try:
    ph.verify(hashed_password, "my_secure_password")
    print("로그인 성공!")
except:
    print("비밀번호 틀림!")

print(f"Hashed Password: {hashed_password}")
```
**👉 특징:**  
✅ **메모리 사용량을 증가시켜 GPU 공격을 어렵게 만듦.**  
✅ bcrypt보다도 강력한 최신 보안 알고리즘.  
✅ 속도가 조금 느릴 수 있지만, 최고의 보안성을 제공.

### 🔴 **레인보우 테이블 공격 과정**  
1️⃣ 해커가 자주 사용되는 비밀번호 리스트(예: `password123`, `qwerty`, `123456`)를 준비함.  
2️⃣ 이 비밀번호들을 해시 함수(SHA-256 같은 것)로 변환해서 `해시값`을 저장함.  
3️⃣ 해킹 대상의 데이터베이스를 훔쳐서, 저장된 해시값과 **미리 계산해둔 값**을 비교함.  
4️⃣ 일치하는 해시값이 있으면, 그에 해당하는 **원래 비밀번호**를 알아낼 수 있음.  

즉, **미리 계산된 해시값 테이블을 이용해서 빠르게 비밀번호를 찾아내는 기법**이야.  
이 방식의 핵심은 **복호화(Decryption)가 아니라, 미리 계산해서 빠르게 찾는 것**이야!  

---

### 🔵 **Salt를 사용하면 레인보우 테이블 공격을 막을 수 있는 이유**
레인보우 테이블은 **"같은 비밀번호는 같은 해시값이 나올 거다"** 라는 원리를 이용하는데,  
각각의 비밀번호에 **Salt(랜덤한 문자열)** 를 추가하면, 해시값이 달라져서 공격이 무력화됨.  

예를 들어  
- `password123 → 해시값: abcdefg123456`  
- `password123 + Salt(랜덤값) → 해시값: zyx987cba654`  

같은 비밀번호라도 해시값이 달라지니까, **해커가 미리 계산한 레인보우 테이블이 무용지물**이 되는 거지.  

---

### 🛠 **레인보우 테이블 공격을 막는 방법**  
✔ **Salt 추가** → 같은 비밀번호라도 매번 다른 해시값이 나옴  
✔ **KDF 사용 (예: PBKDF2, bcrypt, Argon2)** → 연산량을 늘려서 공격을 어렵게 만듦  
✔ **긴 비밀번호 사용** → 해커가 미리 계산하는 걸 불가능하게 만듦  

---

### ✅ **정리**
레인보우 테이블 공격은 **미리 해시값을 계산해서 저장해두고, 빠르게 비밀번호를 찾아내는 방법**  
하지만 **Salt를 추가하면 같은 비밀번호라도 다른 해시값이 나오므로 공격이 무력화됨!**

### ⚠️ **1. RSA는 '복호화'가 가능하다 (비밀번호는 복호화되면 안됨!)**  
- RSA는 **비대칭 암호화 방식**으로,  
  - **공개키(Encryption Key)**로 암호화  
  - **개인키(Decryption Key)**로 복호화할 수 있어.  
- 즉, **해커가 개인키를 탈취하면 모든 비밀번호를 복호화할 수 있음!**  
- 비밀번호는 원래 **일방향(One-Way) 변환**이 되어야 하는데,  
  - RSA는 이를 보장하지 않기 때문에 안전하지 않아.  

✅ **대안:**  
- **단방향 해시 함수(SHA-256, bcrypt, PBKDF2 등) 사용**  
  → 해시 함수는 원본을 알아낼 수 없고, brute-force(무차별 대입) 공격만 가능  
  → salt 추가로 brute-force 공격도 어렵게 만들 수 있음  

---

### ⚠️ **2. RSA는 계산량이 많아서 느림 (성능 문제)**  
- RSA는 **큰 소수(prime number) 연산을 기반으로 동작**하기 때문에,  
  - **비밀번호 저장처럼 수많은 요청이 들어오는 작업에 비효율적**이야.  
- 예를 들어, **수천 명이 로그인할 때마다 RSA 복호화를 수행해야 한다면 서버 부하가 심해짐**  
- 해시 함수는 **가볍고 빠르며, 저장만 하면 비교적 쉽게 검증 가능**  

✅ **대안:**  
- **단방향 해시 함수(SHA-256, bcrypt 등) 사용**  
  → 연산량이 적어 저장/검증 속도가 훨씬 빠름  

---

### ⚠️ **3. RSA는 저장 데이터 크기가 크다 (용량 문제)**  
- RSA 암호화된 데이터는 **비교적 긴 길이의 암호문을 생성**하기 때문에,  
  - **비밀번호 저장 공간이 더 많이 필요함**  
  - 예를 들어, 2048비트 RSA를 사용할 경우 **256바이트** 이상의 암호문이 생성됨  
  - 반면, **SHA-256은 고정된 32바이트(256비트)**로 저장됨  

✅ **대안:**  
- 해시 함수(SHA-256, bcrypt)는 **고정된 크기(예: 256비트)로 저장할 수 있어 효율적**  

---

### 🔥 **결론: 왜 RSA보다 단방향 해시 함수가 나은가?**  
| 비교 항목  | RSA (비대칭 암호화) | 단방향 해시 함수 (SHA-256, bcrypt 등) |
|-----------|------------------|----------------------------|
| **복호화 가능 여부** | O (개인키가 있으면 복호화 가능) | X (복호화 불가능) |
| **계산량(속도)** | 느림 (큰 소수 연산) | 빠름 (단순 연산) |
| **저장 크기** | 큼 (2048비트 RSA → 256바이트 이상) | 작음 (SHA-256 → 32바이트) |
| **보안성** | 개인키가 유출되면 위험 | salt를 추가하면 더 안전 |

💡 **따라서, 비밀번호 저장에는 RSA보다 단방향 해시 함수가 더 적합하다!**  
➡ **RSA는 데이터 전송(암호화/복호화)에 적합**하지만,  
➡ **비밀번호 저장에는 단방향 해시 함수(SHA-256, bcrypt, PBKDF2 등)가 적합**!

### 🔢 **RSA에서 '큰 소수 연산'이란?**  

RSA 암호화는 **매우 큰 소수(Prime Number)** 를 사용하여 암호화를 수행하는 방식이야.  
이때 핵심이 되는 연산이 바로 **모듈러 거듭제곱 연산(modular exponentiation)** 이고,  
이 연산이 **비밀번호 저장에 부적합한 이유 중 하나**가 돼.  

---

### 🧮 **RSA의 핵심 연산 과정 (간단한 예시)**  
RSA에서 가장 중요한 연산은 **다음과 같은 형태**야:  

\[
C = M^e \mod N
\]

- \( C \) = 암호화된 값 (Ciphertext)  
- \( M \) = 평문 (Message, 예: 비밀번호)  
- \( e \) = 공개키의 지수 (Exponent)  
- \( N \) = 두 개의 큰 소수 \( p, q \) 의 곱 (RSA 모듈러스)  

이 연산에서 **"큰 소수 연산"** 이 포함되는 부분이 바로 **\( N \) 값 계산**이야.  

---

### 🏗️ **예제: 작은 숫자로 RSA 암호화 과정**  
(실제 RSA는 2048비트 이상의 소수를 사용하지만, 이해를 쉽게 하려고 작은 숫자로 예시를 들게)  

#### ① 두 개의 소수 선택  
   - \( p = 61 \), \( q = 53 \) → 두 개의 큰 소수  
   - \( N = p \times q = 61 \times 53 = 3233 \)  

#### ② 공개키 선택  
   - 지수 \( e = 17 \) (일반적으로 작은 소수)  
   - 공개키 \( (e, N) = (17, 3233) \)  

#### ③ 암호화 (모듈러 거듭제곱)  
   - 비밀번호 \( M = 65 \) (예제 평문)  
   - 암호문 계산:  
     \[
     C = 65^{17} \mod 3233
     \]
   - 실제 계산 과정:  
     \[
     65^{17} = 621485037674556306798740070688267059265
     \]
     이 값을 \( 3233 \)으로 나눈 나머지를 구하면,  
     \[
     C = 2790
     \]
   - **즉, 암호화된 값 \( C = 2790 \)**  

#### ④ 복호화 (개인키 사용)  
   - 개인키 \( d = 2753 \) (특정 알고리즘으로 구한 값)  
   - 복호화 과정:  
     \[
     M = C^d \mod N = 2790^{2753} \mod 3233
     \]
   - 다시 계산하면,  
     \[
     M = 65
     \]
   - 즉, 원래 평문이 복구됨.  

---

### 🚀 **RSA에서 '큰 소수 연산'이 느린 이유**  
1. **거듭제곱 계산이 매우 크다**  
   - 예제에서는 작은 숫자로 했지만,  
   - **실제 RSA에서는 2048비트(약 600자리) 소수를 사용**  
   - \( N \) 값도 600자리 숫자이므로, 거듭제곱 연산이 **매우 오래 걸림**  

2. **모듈러 연산이 추가로 필요하다**  
   - 단순히 큰 숫자를 계산하는 게 아니라, **모듈러 연산(%)도 반복**해야 함  
   - 효율적인 계산을 위해 `빠른 거듭제곱 (Exponentiation by Squaring)` 같은 알고리즘을 사용하지만,  
   - 그래도 **비밀번호 저장처럼 자주 수행되는 작업에 적합하지 않음**  

---

### 🎯 **결론: 단방향 해시 함수 vs RSA**  
| 비교 항목  | RSA (비대칭 암호화) | 단방향 해시 함수 (SHA-256, bcrypt 등) |
|-----------|------------------|----------------------------|
| **연산 방식** | 큰 소수 연산 (거듭제곱 + 모듈러 연산) | 단순한 해시 계산 |
| **속도** | 느림 (큰 수 연산) | 빠름 (고정된 연산) |
| **복호화 가능 여부** | O (개인키가 있으면 복호화 가능) | X (일방향, 복호화 불가능) |
| **비밀번호 저장 적합성** | X (비효율적) | ✅ 적합 (빠르고 안전) |

💡 **결론: RSA는 비밀번호 저장에 부적합!**  
➡ **해싱(SHA-256, bcrypt, PBKDF2 등)이 비밀번호 저장에 더 효율적이고 안전함!**

### **RSA에서 \( N \)이 왜 튀어나오냐?**  
RSA 암호화에서 **\( N \)**은 **공개키와 비밀키의 핵심 요소**야.  

#### **📌 1. \( N \)이란?**
- 두 개의 큰 소수 \( p \), \( q \)를 곱한 값  
  \[
  N = p \times q
  \]
- 이 \( N \)은 **공개키와 비밀키 둘 다에서 사용됨**.  

---

### **🔑 2. 공개키와 비밀키에 \( N \)이 어떻게 사용되나?**
#### **(1) 공개키 (\( e, N \))**
- \( N \)과 **공개 지수 \( e \)**를 조합하여 공개키를 만듦.  
  \[
  \text{공개키} = (e, N)
  \]
- 모든 사람이 알 수 있음.  

#### **(2) 비밀키 (\( d, N \))**
- \( N \)과 **비밀 지수 \( d \)**를 조합하여 비밀키를 만듦.  
  \[
  \text{비밀키} = (d, N)
  \]
- \( d \)는 소수 \( p, q \)를 사용하여 계산됨.  

---

### **📌 3. RSA 암호화에서 \( N \)이 하는 역할**
#### **(1) 암호화 (Encryption)**
- 평문 \( M \)을 암호문 \( C \)로 변환  
  \[
  C = M^e \mod N
  \]
- \( N \)이 없으면 모듈러 연산을 못 함.  

#### **(2) 복호화 (Decryption)**
- 암호문 \( C \)를 복호화하여 원래의 \( M \)을 얻음  
  \[
  M = C^d \mod N
  \]
- \( N \)이 없으면 해독 불가능.  

---

### **📌 4. 왜 \( N \)이 중요한가?**
- **\( N \)은 공개되지만, 소수 \( p, q \)는 비밀**  
- 만약 공격자가 \( N \)을 보고 \( p, q \)를 찾아내면 RSA가 깨짐!  
- 큰 소수를 사용해야 \( p, q \)를 **소인수분해하기 어려움**.  

---

### **📌 5. 정리**
✔ **\( N = p \times q \)**는 RSA 암호화에서 핵심 요소  
✔ 공개키 **(\( e, N \))**와 비밀키 **(\( d, N \))** 모두에서 사용됨  
✔ \( N \)이 없다면 암호화·복호화 계산이 불가능  
✔ \( p, q \)를 모르면 \( N \)만 가지고는 비밀키를 알 수 없음  

➡ **즉, \( N \)은 공개되어 있지만, 안전성은 \( p, q \)의 비밀 유지에 달려 있음!**

```text
- Public Key: 그 사람의 공개키
- Issuer: 누가 발급했는지 (CA 이름)
- Valid From / To: 유효기간
- Signature: CA가 이 인증서에 서명한 것
```

- **공개키**: 너의 자물쇠 (누구나 이걸로 편지 암호화 가능)
- **비밀키**: 너만 갖고 있는 열쇠 (편지 복호화 가능)

## 🔐 비밀키(Private Key)가 중요한 이유
- **비밀키가 탈취되면** 문제가 생김:
  - 해커가 **진짜 서버인 척 위장(피싱)** 가능
  - **디지털 서명 위조** 가능

- **HSM(Hardware Security Module)** 같은 장비에 보관
- 또는 **암호화된 저장소에 잠금 + 접근 제한** 걸어둠

## ✍️ ② 비밀키로 서명 / 공개키로 서명 검증 – "진짜임을 증명"

### 📦 상황: 서버가 클라이언트에게 보낸 메시지가 **진짜 서버가 보낸 것**임을 보장하고 싶음

1. 서버는 메시지를 만들고,  
   메시지에 대해 **해시값**을 계산함

2. 그 해시값을 **비밀키로 서명**함  
   → 서명된 해시값 = 전자서명

3. 클라이언트에게 메시지 + 서명 전달

4. 클라이언트는:
   - 메시지에 대해 똑같이 해시값 계산
   - 서명을 **공개키로 복호화**해서 비교

5. 둘이 일치하면:
   ✅ → 진짜 서버가 보낸 거구나!

🧠 요점:  
- **서버만 비밀키로 서명 가능**  
- **누구든 공개키로 서명 검증 가능**  
→ ▶ **신원 보장**

## 🔐 PKI 구조가 어떻게 막는지?

### ✅ 인증서 구성 (X.509 형식)

```text
Certificate {
  Subject: secure.site.com
  PublicKey: ABCD1234...
  Issuer: DigiCert
  Signature: IssuerPrivateKey로 서명된 값
}
```

### ✅ 검증 과정

1. **서버가 인증서를 보냄**
2. 브라우저는 인증서 안의 "Subject" 값을 보고 → **도메인 이름과 일치하는지 검사**
3. "Issuer" 값을 보고 → OS/브라우저에 등록된 **신뢰할 수 있는 CA 목록**에서 찾음
4. 인증서의 서명을 보고 → **CA의 공개키**로 서명을 검증함
5. 위 과정이 실패하면, **"이 사이트는 안전하지 않음" 경고** 발생

---

## ❌ 위조가 불가능한 이유

| 시도 | 왜 실패하는가 |
|------|----------------|
| 인증서 내용을 복붙함 | 도메인 이름(`Subject`)가 다르면 **브라우저가 경고** |
| 인증서 내용을 조작함 | 서명 값이 달라져서 **서명 검증 실패** |
| 자체 인증서 발급 | 브라우저가 신뢰하지 않는 **루트 CA** → 경고창 |
| 피싱 사이트에서 똑같이 생긴 인증서 사용 | 도메인이 다르면 무조건 실패 |

➡️ **PKI 기반의 인증 체계**를 따릅니다.

그리고 **SSL/TLS처럼 세션키(대칭키)**도 같이 써요.  
즉, **공개키(비대칭)** + **세션키(대칭)** 조합으로 동작하는 건 맞습니다.

🔐 **요약:**
- 공개키 기반의 인증서를 사용해 **세션키를 안전하게 주고받고**
- 이후에는 **세션키로 빠르게 통신**하는 구조

## ❓그럼 SSL은 왜 이제 안 써?

### ▶ SSL은 TLS의 **이전 버전**이에요.

| 버전 | 이름 |
|------|------|
| SSL 1.0, 2.0, 3.0 | 초기 버전 (1990년대) → **보안 취약** |
| TLS 1.0, 1.1, 1.2, 1.3 | 최신 버전 (2000년대~현재) |

---

### ❌ SSL이 퇴출된 이유
1. **암호화 알고리즘에 취약점** 많음 (예: POODLE, BEAST 공격 등)
2. **핸드셰이크 구조가 느리고 비효율적**
3. **현대 보안 수준을 충족하지 못함**

그래서 현재는 거의 모든 브라우저/서버가 **TLS 1.2 또는 1.3**만 사용하고,
SSL은 완전히 퇴출되었어요.

## 🔌 SSL/TLS는 뭐냐면?

SSL이나 TLS는 그냥 **"안전하게 통신하기 위한 규약(프로토콜)"**이에요.

- 보통 우리가 인터넷에서 정보를 주고받을 때는 TCP 소켓을 써요.  
- 그런데 이건 **암호화가 안 됨**. 누가 가로채면 끝이에요.
- 그래서 거기 **위에 덧붙여지는** 게 **SSL/TLS**입니다.

➡️ **TCP/IP 위에서 작동하는 보안 계층**이란 뜻이에요.  
소켓 자체는 단순한 통신 채널이고, TLS는 그걸 안전하게 만드는 껍데기라고 보면 돼요.

---

## 🧱 계층 구조 간단 예시:

```
[Application Layer] ← 웹, 메일 등
[SSL/TLS Layer]     ← 이게 보안 추가해줌 (대칭키 + 인증서 등)
[Transport Layer]   ← TCP 소켓
[Network Layer]     ← IP
```

즉, **TCP 소켓으로 연결**하고 → **TLS로 인증/암호화**해서 → **HTTP나 다른 데이터를 안전하게 보내는** 구조.

---

## 📦 그래서 HTTPS는?

- `HTTP` + `TLS` = `HTTPS`
- 웹에서 HTTP 요청을 보내기 전에 **TLS Handshake**로 세션 키와 인증서 등을 주고받고,
- 그 후 모든 통신은 TLS로 **암호화된 상태로 진행**돼요.

---

## ✨ 정리 요약

| 용어 | 설명 |
|------|------|
| **소켓(Socket)** | 그냥 통신 파이프 (암호화 없음) |
| **SSL/TLS** | 이 소켓 위에서 안전하게 통신하게 해주는 보안 계층 |
| **HTTPS** | TLS가 적용된 HTTP |

## 🕹️ 실시간 게임에서의 네트워크 통신 방식:

### 🔸 1. **UDP 기반 통신**
- 대부분 실시간 게임은 **UDP (User Datagram Protocol)** 사용.
- 이유:  
  - **빠름** (TCP보다 훨씬 가볍고 빠름)  
  - **순서 보장 X, 신뢰성 X → 그 대신 속도 O**
- 게임에서는 "0.1초 전에 맞은 총알"을 굳이 다시 받을 필요가 없거든요.
  - 대신 **매 프레임마다 최신 정보를 보내서 갱신**함.

> 예: “플레이어 A 위치 x=123, y=456” 같은 걸 계속 보냄.

---

### 🔸 2. **TLS는 안쓰냐?**
- **게임 로그인 과정**이나 **계정/결제 정보 전달**에는 TLS(HTTPS 등)를 써요.  
  - 예: 롤 클라이언트에서 로그인할 때는 **TLS로 암호화된 HTTPS 요청**을 보냄.
- BUT! **인게임 중에는 TLS 같은 무거운 프로토콜은 거의 안 씀.**
  - 프레임 지연이 발생하면 게임성에 영향이 큼.

---

### 🔸 3. **암호화는 어떻게 하냐?**
- 실시간 통신은 보통 **TLS 말고 자체 경량 암호화** 방식을 써요.
  - 예: 대칭키 기반의 경량 알고리즘 (예: AES-GCM 또는 자체 제작)
  - 로그인할 때 받은 세션 키를 **UDP 패킷 암호화에 사용**하기도 해요.
- 게임 서버와 클라이언트는 **초기에 키 교환**을 하고, 이후엔 대칭키로 실시간 암호화.

---

### 🔸 4. 예시 흐름 (롤 기준 추정)

1. 클라이언트 → 로그인 서버 접속 (HTTPS / TLS)
2. 사용자 인증 → 세션 토큰, 키 수신
3. 매치메이킹 → 게임 서버 할당
4. 인게임 진입 → **UDP 포트로 통신 시작**
5. 게임 서버와 **암호화된 UDP 패킷**으로 계속 주고받음
6. 끝나면 결과를 다시 HTTPS로 전송

| 인게임 | UDP + 경량 암호화 | 속도 중요, 약간의 손실 허용 |
| 게임 종료 후 결과 | HTTPS | 다시 안전하게 처리 |

## 📌 TLS 핸드셰이크 요약

HTTPS나 TLS를 사용하는 웹사이트에 접속할 때, 브라우저(클라이언트)와 서버가 **처음 연결을 시작할 때 수행하는 절차**입니다.

---

## ✅ 기본 흐름 (대칭키 기반 세션을 만들기 위한 공개키 기반 핸드셰이크)

```
클라이언트                          서버
   |         ClientHello            |
   |------------------------------->|
   |                                |
   |         ServerHello            |
   |   + 인증서(Certificate)       |
   |   + 공개키                     |
   |<-------------------------------|
   |                                |
   |   (공개키로 세션 키 암호화)    |
   |   + Encrypted Pre-Master Key  |
   |------------------------------->|
   |                                |
   |     세션 키를 통해 암호화 준비 완료 |
   |<------------------------------>|
   |      대칭키로 안전한 통신 시작     |
```

---

## 🧩 각 단계 설명

| 단계 | 설명 |
|------|------|
| **1. ClientHello** | 클라이언트가 서버에 "나 TLS 쓸거야!" 하며 접속. |
| **2. ServerHello + 인증서** | 서버가 "좋아, 이게 내 인증서야(공개키 포함)" 하며 보냄. |
| **3. 키 교환** | 클라이언트가 공개키로 암호화된 **Pre-Master Key**를 보냄. |
| **4. 세션 키 생성** | 서버는 개인키로 복호화하여 세션 키를 생성. 양측 모두 **같은 대칭키** 확보. |
| **5. 이후 통신은 대칭키로** | 빠르고 가볍게, 대칭키로 암호화된 안전한 통신 시작. |

---

## 🤔 왜 이렇게 복잡하게 하나요?

- **공개키 암호화**는 느리지만 안전 → **처음 연결할 때만 사용**  
- **대칭키 암호화**는 빠름 → **세션 키로 이후 통신에 사용**

이렇게 하면 **속도와 보안**을 모두 챙길 수 있어요.

---

## ✨ 결과적으로

- 클라이언트와 서버가 **"서로 신뢰"**하고
- **같은 세션 키**를 갖고
- **암호화된 통신 준비 완료**

이게 **핸드셰이크 완료** 상태입니다!

### ✅ 1. `HTTP` vs `HTTPS`  
| 구분 | HTTP | HTTPS |
|------|------|-------|
| 의미 | HyperText Transfer Protocol | **HTTP + TLS(SSL)** 조합 |
| 암호화 | 없음 (평문 전송) | 있음 (데이터 암호화됨) |
| 보안성 | 낮음 | **높음 (보안 통신)** |
| 사용 예 | 내부 테스트, 구형 시스템 | **모든 보안이 필요한 웹사이트 (로그인, 결제 등)** |
| 주소창 표시 | http:// | https:// (자물쇠 아이콘) |

**즉, HTTPS는 “HTTP에 TLS를 입힌 것”이라고 보면 됩니다.**

---

### ✅ 2. TLS(전 SSL)는 뭐냐?

| 항목 | 설명 |
|------|------|
| 풀네임 | **Transport Layer Security** |
| 역할 | **웹 통신을 암호화**하는 보안 프로토콜 |
| 이전 이름 | SSL (Secure Sockets Layer), 현재는 거의 다 TLS 사용 |
| 위치 | HTTP 아래의 **전송 계층**에서 동작 |
| 주요 기능 | 암호화, 인증서 검증, 세션키 교환 등 |

즉, TLS는 **보안 통신을 위한 규칙(프로토콜)**이고,  
HTTPS는 그 TLS를 **HTTP 위에 얹어서 사용하는 형태**입니다.

> **HTTPS = HTTP + TLS**

## ✅ 핵심 정리

---

## 📌 단계별로 보면?

### 1. **개념적 설계 (ER 다이어그램)**
- 실제 세계의 개체(Entity), 속성(Attribute), 관계(Relationship)를 도식화함.
- 예: `학생`은 `이름`, `학번`, `반`이라는 속성을 가지고 있고, `반`이라는 개체와 관계를 맺음.

### 2. **논리적 설계**
- ER 다이어그램을 **릴레이션 스키마로 변환**.
  - 개체 → 테이블
  - 속성 → 컬럼
  - 관계 → 외래 키 등 제약조건
- 정규화를 통해 구조를 정돈

### 3. **물리적 설계**
- 실제 DBMS에 맞게 구현 가능한 설계로 변환
- 인덱스, 파티셔닝, 스토리지 세부 설정 등 추가 고려
- SQL로 테이블 생성할 준비 단계

---

## 🧩 예시

ER 다이어그램에:

```
학생(Student) —[속성: 학번, 이름]  
수업(Course) —[속성: 과목코드, 과목명]  
학생은 수업을 수강한다 (관계)
```

이걸 논리적 설계로 바꾸면:

```sql
Student(학번 PK, 이름)
Course(과목코드 PK, 과목명)
Enrollment(학번 FK, 과목코드 FK)
```

그리고 이걸 실제 물리적 설계로:

```sql
CREATE TABLE Student (
  id INT PRIMARY KEY,
  name VARCHAR(50)
);

CREATE TABLE Course (
  code VARCHAR(10) PRIMARY KEY,
  title VARCHAR(100)
);

CREATE TABLE Enrollment (
  student_id INT,
  course_code VARCHAR(10),
  FOREIGN KEY (student_id) REFERENCES Student(id),
  FOREIGN KEY (course_code) REFERENCES Course(code)
);
```

## 🔹 `GROUP BY CUBE(A, B)`란?

`CUBE`는 다차원 집계를 위한 SQL 기능이에요.  
`GROUP BY A, B`는 딱 한 조합으로만 그룹핑하지만,  
`**CUBE(A, B)**`는 **가능한 모든 조합의 집계를 자동 생성**합니다.

---

### 🔸 `GROUP BY CUBE(A, B)`가 만드는 집계 조합

| 그룹핑 A | 그룹핑 B | 설명 |
|----------|----------|------|
| A        | B        | A와 B 각각으로 그룹 |
| A        | NULL     | B 전체에 대한 A별 집계 |
| NULL     | B        | A 전체에 대한 B별 집계 |
| NULL     | NULL     | 전체 집계 (총합 등) |

✅ 총 **4가지 조합**이 만들어짐

---

### 📝 정답

**O (맞음)**

👉 `GROUP BY CUBE(A, B)`는 **총 2^n = 2² = 4개**의 조합을 생성함.

좋아요! `GROUP BY CUBE(A, B)`를 쓰는 실제 예시 하나 들어볼게요.

---

### 📘 예제 테이블: `sales`

| region | product | sales |
|--------|---------|-------|
| East   | Pen     | 100   |
| East   | Paper   | 150   |
| West   | Pen     | 200   |
| West   | Paper   | 300   |

---

### 🔹 쿼리: `GROUP BY CUBE(region, product)`

```sql
SELECT region, product, SUM(sales) AS total_sales
FROM sales
GROUP BY CUBE(region, product);
```

---

### 🔸 결과:

| region | product | total_sales |
|--------|---------|-------------|
| East   | Pen     | 100         |
| East   | Paper   | 150         |
| West   | Pen     | 200         |
| West   | Paper   | 300         |
| East   | NULL    | 250         -- East 전체 제품 매출 |
| West   | NULL    | 500         -- West 전체 제품 매출 |
| NULL   | Pen     | 300         -- Pen 전체 지역 매출 |
| NULL   | Paper   | 450         -- Paper 전체 지역 매출 |
| NULL   | NULL    | 750         -- 전체 매출 합계 |

---

### 🧠 요약

- `CUBE(region, product)`는:
  - 지역별 *제품* 매출
  - 지역별 *전체* 제품 매출
  - 전체 지역의 *제품* 매출
  - **전체 합계**까지 한 번에 나와서
- 비즈니스 인사이트(예: "어느 지역에서 펜이 잘 팔렸지?" vs "총 매출은?")를 빠르게 뽑아볼 수 있어요.
