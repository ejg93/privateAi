`<E>` 같은 표현은 **제네릭(Generic)** 이라고 부르는 Java 문법이에요.  
보통 클래스나 메서드에서 **타입을 일반화**해서 여러 자료형을 유연하게 다룰 수 있도록 할 때 사용합니다.  

---

### 1. 제네릭 클래스
```java
public class Box<E> {
    private E value;

    public void set(E value) {
        this.value = value;
    }

    public E get() {
        return value;
    }
}
```
- `Box<Integer>` → `E`가 `Integer`로 치환됨  
- `Box<String>` → `E`가 `String`으로 치환됨  

```java
Box<Integer> intBox = new Box<>();
intBox.set(123);
System.out.println(intBox.get()); // 123

Box<String> strBox = new Box<>();
strBox.set("Hello");
System.out.println(strBox.get()); // Hello
```

---

### 2. 제네릭 메서드
```java
public class Util {
    public static <E> void printElement(E element) {
        System.out.println(element);
    }
}
```

- `E`라는 타입 변수를 메서드 선언부에 붙여두면, 호출할 때 타입이 자동으로 결정됨.
```java
Util.printElement(100);     // E → Integer
Util.printElement("Java");  // E → String
```

---

### 3. 여러 타입 파라미터
```java
public class Pair<K, V> {
    private K key;
    private V value;

    public Pair(K key, V value) {
        this.key = key;
        this.value = value;
    }

    public K getKey() { return key; }
    public V getValue() { return value; }
}
```

```java
Pair<String, Integer> pair = new Pair<>("Age", 20);
System.out.println(pair.getKey());   // Age
System.out.println(pair.getValue()); // 20
```

---

✅ 요약  
- `<E>` → 타입 파라미터 (보통 **Element** 의미)  
- `<T>`, `<K>`, `<V>` 등 다른 글자도 많이 씀 (의미만 다를 뿐 동작은 같음)  
- **컴파일 타임에 타입 안정성 보장** + **중복 코드 제거**

## 🔹 제네릭이 왜 좋은가?
Java는 **정적 타입 언어**라서, 변수의 타입이 **컴파일 시점**에 정해져야 합니다.  
제네릭은 여기서 **유연성과 안전성**을 동시에 챙기려는 장치예요.

예를 들어, 제네릭이 없던 시절(`Java 5 이전`)에는 이렇게 코딩했어요:
```java
List list = new ArrayList();
list.add("Hello");
list.add(123); // 문자열 + 숫자 섞임 (컴파일러는 문제 삼지 않음)

String s = (String) list.get(0); // 직접 캐스팅
```

⚠️ 문제: `list.get(1)`에서 `Integer`를 `String`으로 캐스팅하려 하면 런타임 에러 발생.  
→ 버그는 실행해야만 드러남.  

---

제네릭을 쓰면:
```java
List<String> list = new ArrayList<>();
list.add("Hello");
// list.add(123);  // 컴파일 에러! 방지됨

String s = list.get(0); // 안전, 캐스팅 불필요
```

👉 즉, **제네릭은 “실행 전에” 잘못된 타입을 잡아주는 안전망** 역할을 해요.  

---

## 🔹 그럼 언제 쓰는 게 맞을까?

### ✅ 제네릭이 적합한 경우
1. **컬렉션, 자료구조**  
   - `List<E>`, `Map<K,V>` 같은 경우  
   - 여러 타입으로 쓰이되, 하나의 인스턴스에서는 **일관된 타입 보장** 필요할 때
   - 예: `List<String>` / `List<Integer>`

2. **재사용 가능한 유틸 코드**  
   - 타입마다 따로 코드 복붙하기 싫을 때  
   ```java
   public static <T> T pickRandom(List<T> list) { ... }
   ```

3. **라이브러리/프레임워크 API 설계**  
   - 사용자 코드와 타입이 맞물리도록 강제하고 싶을 때
   - 예: Spring의 `ResponseEntity<T>`

---

### ❌ 굳이 안 써도 되는 경우
1. **타입이 딱 정해져 있는 로직**  
   - 예: `UserRepository`는 무조건 `User`만 다룬다 → 굳이 `<E>` 필요 없음.

2. **즉흥적인 스크립트/단순 데이터 처리**  
   - JS처럼 빠르게 코딩하는 상황에서는 제네릭/타입 강제가 오히려 “귀찮은 보일러플레이트”가 될 수 있음.

---

## 🔹 JS ↔ TS ↔ Java와 비교
- **JS** : 타입 자유롭지만, 런타임 오류가 잦음  
- **TS** : 타입 시스템 덕분에 오류를 미리 잡을 수 있지만, 너무 엄격하면 답답하다고 느껴서 다시 JS로 돌아가는 사람도 있음  
- **Java** : 애초에 정적 타입 언어 → 제네릭은 타입 안정성과 재사용성을 위한 “선택적 완충 장치”  

---

✅ **정리**  
- 제네릭은 “내 코드가 여러 타입에서 쓰일 수 있는데, 타입 안정성도 보장하고 싶다” 할 때 사용.  
- “타입이 명확히 고정된 상황”에서는 오히려 불필요한 복잡성.  
- 그래서 **API, 공용 유틸, 자료구조**에는 잘 맞고, **도메인 특화 코드**에는 굳이 필요 없어요.

## 🔹 `Object` 사용하는 방식
```java
List list = new ArrayList();   // 제네릭 안 씀
list.add("Hello");
list.add(123);  // 가능

Object o = list.get(0);
String s = (String) o;   // 직접 캐스팅해야 함
```

- `Object`는 모든 클래스의 최상위 타입 → 어떤 타입이든 다 넣을 수 있음  
- 하지만 **꺼낼 때 무조건 캐스팅 필요**  
- 잘못된 캐스팅은 런타임 오류 발생  

---

## 🔹 제네릭 사용하는 방식
```java
List<String> list = new ArrayList<>();
list.add("Hello");
// list.add(123); // 컴파일 에러!

String s = list.get(0); // 캐스팅 불필요
```

- 제네릭은 컴파일 시점에 타입 체크 가능  
- **안전하고 가독성도 좋아짐**  
- 캐스팅을 없애면서도, 다양한 타입을 지원 가능  

---

## 🔹 차이를 한눈에 비교

| 구분 | `Object` 사용 | 제네릭 사용 |
|------|---------------|-------------|
| **저장 가능 타입** | 모든 타입 가능 | 지정한 타입만 가능 |
| **컴파일 시점 오류** | 없음 (런타임 오류 위험 ↑) | 있음 (안전성 ↑) |
| **꺼낼 때** | 캐스팅 필요 | 캐스팅 불필요 |
| **가독성** | 타입 정보 알기 힘듦 | 타입 정보 명확 |
| **재사용성** | 비슷 | 더 명확하고 안정적 |

---

## 🔹 비유로 설명해보면
- `Object` 방식 → **“큰 박스”** : 아무거나 다 넣을 수 있지만, 꺼낼 때 내가 뭘 넣었는지 기억해야 하고 잘못 꺼내면 터짐.  
- 제네릭 방식 → **“라벨 붙은 박스”** : “이 박스엔 사과만 들어간다”라고 정해져 있어서, 실수로 바나나를 넣으려 하면 애초에 안 들어감.  

---

✅ 정리  
- **제네릭은 `Object`를 대신하는 더 안전하고 가독성 좋은 방식**  
- `Object` 방식은 유연하지만, 런타임 오류 위험이 크고, 캐스팅이 필수라서 코드가 지저분해짐  
- 그래서 컬렉션 같은 데서는 `Object` 대신 **제네릭**이 사실상 표준

---

### 🔹 오해하기 쉬운 포인트
```java
List<String> list = new ArrayList<>();
```
- 여기서 이미 `list`라는 변수는 **"문자열만 담는 리스트"** 라는 계약(타입 파라미터)을 가진 상태예요.  
- 따라서 첫 번째 `add()` 호출과 상관없이 **애초에 `Integer` 같은 다른 타입은 들어올 수 없음**.  

즉, **“하나라도 들어가면 그때부터 고정”** 이 아니라,  
**“타입 파라미터를 선언할 때 이미 고정”** 되는 거예요.  

---

### 🔹 비교해볼 예시

#### (1) 제네릭 사용
```java
List<String> list = new ArrayList<>();
list.add("Hello");   // 가능
list.add(123);       // ❌ 컴파일 에러 (Integer는 String 아님)
```

#### (2) 제네릭 미사용 (`Object` 방식)
```java
List list = new ArrayList();  // 타입 지정 X
list.add("Hello");   // 가능
list.add(123);       // 가능
```
- 이 경우엔 타입이 고정되지 않음. (Object 취급)  
- 나중에 꺼낼 때 `String s = (String) list.get(1);` 하면 런타임 오류 발생 가능.

---

### 🔹 정리
- `List<E>` → **타입 파라미터 `E`는 선언 시점에 확정됨** (예: `List<String>` → String으로 확정)  
- “첫 요소로 타입이 정해진다”가 아니라, **변수 선언부/객체 생성부에서 정하는 것**  
- 그래서 제네릭은 **타입 안정성을 컴파일 단계에서 강제**할 수 있는 거예요.
