# AI 활용 메모

> 개인 정리 노트. AI 쓰면서 배운 것 누적. 날짜 달고 추가할 것.

## 2026-07-10 — AI 코드 리뷰 제대로 시키는 법

우선순위: **백지 컨텍스트 > 구체적 공격 지시 > 모델 다양성**. 페르소나 연기는 버려도 됨.

### 1. 독립 컨텍스트 (효과 제일 큼)

- 작성자의 대화 이력을 물려받은 리뷰어는 작성자 가정을 그대로 믿음 — "아까 검증했다며" 앵커링.
- 같은 모델이라도 **백지에서 시작**시키는 게 다른 모델 쓰는 것보다 효과 큼.
- 실전: fork(대화 이력 상속) 말고 fresh 에이전트로 띄울 것.

### 2. 과제 프레이밍 (여전히 필요, 단 옛날 방식 아님)

- "너는 20년차 시니어 보안 전문가야" 식 페르소나 부여는 이제 거의 플라시보.
  요즘 모델은 정체성 연기가 아니라 **구체적 지시**에 반응함.
- 필요한 3요소:
  1. **결함 클래스 명시** — "따옴표 입력 추적해라", "방언별 딕셔너리 뷰명 검증해라"
  2. **공격 관점** — "깨지는 입력 찾아라"
  3. **출력 형식** — "결함만, 기능 제안 금지, 심각도 표기"
- 역할 지정이 의미 있는 유일한 경우: 리뷰어 여럿 돌릴 때 **범위 분담**
  (A는 JS만, B는 SQL만) — 겹침 줄이고 깊이 확보.

### 3. 다른 모델 (도움되지만 3순위)

- 학습이 다르면 사각지대가 다름. 단, 같은 회사 모델끼리(Fable↔Opus)는 편향 일부 공유.
- 다양성 최대치는 타 벤더 (ChatGPT/Gemini).
- 1·2번 없이 모델만 바꾸면 효과 미미.

### 4. 결과 중재 (다들 빼먹는 것)

- 리뷰어 늘리면 오탐도 같이 늘어남.
- 나온 지적을 그대로 믿지 말고 **코드 대조 검증 후 반영**.
- 리뷰 시킨 AI와 검증하는 AI를 분리하거나, 최종은 사람이 코드 보고 판단.
