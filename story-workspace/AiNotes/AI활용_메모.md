# AI 활용 메모

> 개인 정리 노트. AI 쓰면서 배운 것 누적. 날짜 달고 추가할 것.

## 2026-07-10 — AI 코드 리뷰 제대로 시키는 법

우선순위: **백지 컨텍스트 > 구체적 공격 지시 > 모델 다양성**. 페르소나 연기는 버려도 됨.

### 1. 독립 컨텍스트 (효과 제일 큼)

- 작성자의 대화 이력을 물려받은 리뷰어는 작성자 가정을 그대로 믿음 — "아까 검증했다며" 앵커링.
- 같은 모델이라도 **백지에서 시작**시키는 게 다른 모델 쓰는 것보다 효과 큼.
- 실전: fork(대화 이력 상속) 말고 fresh 에이전트로 띄울 것.

### 2. 과제 프레이밍 (여전히 필요, 단 옛날 방식 아님)

- "너는 20년차 시니어 보안 전문가야" 식 페르소나 부여는 이제 거의 플라시보.
  요즘 모델은 정체성 연기가 아니라 **구체적 지시**에 반응함.
- 필요한 3요소:
  1. **결함 클래스 명시** — "따옴표 입력 추적해라", "방언별 딕셔너리 뷰명 검증해라"
  2. **공격 관점** — "깨지는 입력 찾아라"
  3. **출력 형식** — "결함만, 기능 제안 금지, 심각도 표기"
- 역할 지정이 의미 있는 유일한 경우: 리뷰어 여럿 돌릴 때 **범위 분담**
  (A는 JS만, B는 SQL만) — 겹침 줄이고 깊이 확보.

### 3. 다른 모델 (도움되지만 3순위)

- 학습이 다르면 사각지대가 다름. 단, 같은 회사 모델끼리(Fable↔Opus)는 편향 일부 공유.
- 다양성 최대치는 타 벤더 (ChatGPT/Gemini).
- 1·2번 없이 모델만 바꾸면 효과 미미.

### 4. 결과 중재 (다들 빼먹는 것)

- 리뷰어 늘리면 오탐도 같이 늘어남.
- 나온 지적을 그대로 믿지 말고 **코드 대조 검증 후 반영**.
- 리뷰 시킨 AI와 검증하는 AI를 분리하거나, 최종은 사람이 코드 보고 판단.

## 2026-07-13 — 검증시키는 법 (읽히지 말고 돌리게 하라)

**AI에게 코드를 읽히지 말고 실행시켜라.** 읽기는 추측이고 실행은 사실이다.
도구는 이미 다 있어도 시키지 않으면 안 켠다. 켜지게 만드는 게 내 일.

### 1. "체크해줘" 금지 → "테스트 짜서 돌려봐"

- "체크해줘"는 AI를 **평론가**로 쓰는 것. 그럴듯한 지적만 나오고 많이 놓친다.
- "테스트 하네스 짜서 실제로 돌려라"가 AI를 **엔지니어**로 쓰는 것.
- 정규식·파싱·경계값·날짜처럼 **머리로 시뮬레이션하면 틀리는 영역**에서 격차가 제일 크다.
- 회귀 검증이 공짜로 딸려온다. A 고치다 B 깨먹었는지 즉시 확인.

### 2. 하네스 짜는 규칙

1. **순수 함수만 추출해 실행** — 소스에서 로직 함수만 뽑아 node로 로드.
2. **라인번호 하드코딩 금지** — 함수명으로 찾아 슬라이스. 안 그러면 수정 후 어긋난다.
3. **케이스는 "깨지는 입력"으로** — 정상 케이스만 짜면 의미 없음.
   (7/10 메모의 "공격 관점"이 여기 그대로 적용)
4. **OK/BUG 라벨로 기대값 vs 실제값 대조 출력.**
5. **고친 뒤 같은 하네스 재실행.**

### 3. 실행 못 하는 건 공식 문서로 대조

- 실행 수단이 없으면(폐쇄망 DB 등) **공식 문서를 직접 받아서 대조**시켜라.
  PDF도 받아서 파싱 가능. "기억으로 답하지 말고 문서 근거를 대라"고 지시.
- 벤더 문서가 있는지부터 확인. (예: Sybase ASE → SAP이 인수, `help.sap.com`)
- 이것도 안 되면 **미해결로 남겨라.** 억지로 결론 내게 하지 말 것.

### 4. "실행한 것"과 "추론한 것"을 구분해 말하게 하라

- 검증된 것과 그럴듯한 것이 같은 톤으로 나오는 게 **제일 위험**하다.
- 틀린 걸 자신 있게 말하는 것보다 **뭘 모르는지 안 밝히는 게** 손해가 크다.
  (모르는 걸 숨기면 검증된 부분의 신뢰도까지 같이 죽는다)
- AI 스스로도 자기 추론보다 실행 결과를 믿어야 한다.
  실제로 AI가 손으로 계산한 기대값이 틀리고 실행 결과가 맞은 사례 있었음.

### 5. AI가 없는 문제를 만들면 밀어내라

- AI는 **일감을 만들어내는 경향**이 있다. "문제점 3건"이라고 보고했는데
  따져보니 표준을 안 지킨 입력을 가정했거나, 애초에 오류가 아닌 경우가 있다.
- 지적을 받으면 되묻기: **"그게 정상적인 사용에서 실제로 나오는 입력이냐?"**
- 근거가 약하면 그냥 잘라라. 안 그러면 없는 버그 고치느라 시간 날린다.
- 7/10 메모의 "결과 중재"와 같은 원리 — **리뷰어를 리뷰해야 한다.**

### 6. 검증하는 쪽도 검증하라

- **"각 항목 몇 건씩 테스트했냐, 커버리지 균등하냐"**고 물어라.
- AI는 스스로 빠뜨린 걸 먼저 밝히지 않는다. 물어야 나온다.

### 남는 구멍 (알고 쓸 것)

- **순수 함수 테스트는 DOM 배선을 검증 못 한다.** UI 이벤트 버그는 안 잡힌다.
  → 완결하려면 Puppeteer/jsdom으로 실제 페이지 열어 클릭까지 시켜야 함.
- **SQL은 실 DB 없이 실행 검증 불가.** 문서 대조까지가 한계.
  뷰·컬럼 존재는 확인돼도 조인 결과 행 뻥튀기 같은 **런타임 거동은 추론**임.

### 내 환경 (2026-07-13 기준)

| 도구 | 상태 |
|------|------|
| Node 22 | 설치됨 — **AI 검증 주력. 사실상 유일 선택지** |
| JDK 25 | 설치됨. `java Foo.java` 단일파일 실행 가능 (컴파일 불필요) |
| pdftotext | 있음 — 공식 문서 PDF 대조에 씀 |
| Chromium | `~/.cache/puppeteer` 에 있음. puppeteer 패키지만 깔면 브라우저 구동 가능 |
| Python | **안 깔림** (MS Store 스텁만 잡힘 — 실행하면 실패) |
| maven / pip / docker / jq | 없음 |

→ **AI에게 파이썬으로 검증 스크립트 짜라고 하면 안 됨.**
